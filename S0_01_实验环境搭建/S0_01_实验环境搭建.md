# S0_01 — 实验环境搭建

## 1. 本节目标

- 在 Ubuntu 24.04 上安装 QEMU 9.0（支持 PCIe 完整模拟）
- 配置 Linux 内核源码（6.17），能够编译 PCI 驱动模块
- 验证 QEMU virt 平台的 PCIe RC 模拟正常（ECAM + MSI + AER）
- 建立开发目录结构，熟悉实验流程

## 2. 前置依赖

无。本节是所有实验的起点。

## 3. 源码位置

本节不涉及内核 PCI 源码，为纯环境配置。

---

## 4. 为什么这样设计

### 为什么要用 QEMU，不用真实硬件？

| 方案 | 优点 | 缺点 |
|------|------|------|
| **QEMU virt** | PCIe RC 完整模拟，ECAM/MSI/AER 全支持，可改寄存器，可插 EP | 速率是模拟的，不是 Gen3/4 真实速率 |
| **真实 x86 PC** | 真实 PCIe Gen4 | 不能改 Intel RC 驱动，无法探查内部状态 |
| **树莓派 4** | 真实硬件，可写 RC 驱动 | 只有一个端口，调试手段少 |

**选 QEMU virt**：足够完成 S2~S5 所有实验，且可以观察内核 PCI 子系统所有内部状态。

### 为什么用 Ubuntu 24.04？

- GCC 13 / GDB 14 / QEMU 9.0 直接 apt 安装，无需编译
- HWE 内核 6.17，与 headers 完全匹配，避免版本错配

## 5. QEMU 安装步骤

### 5.1 卸载旧版本（如果有）

```bash
sudo apt remove qemu-system-x86 qemu-system-x86_64 qemu-utils --purge -y 2>/dev/null
sudo apt autoremove -y
```

### 5.2 从源码编译安装 QEMU 9.0

Ubuntu 24.04 官方源只有 QEMU 8.2，PCIe SRS 特性（Alternative Routing-ID Forwarding）在 8.2 有 bug，**必须用 QEMU 9.0+**。

```bash
# 安装编译依赖
sudo apt update
sudo apt install -y \
  build-essential pkg-config libglib2.0-dev libfdt-dev \
  libpixman-1-dev libncurses-dev git cmake \
  libssl-dev libaio-dev libusb-1.0-0-dev \
  libsdl2-2.0-0 libsdl2-image-2.0-0

# 下载 QEMU 9.0
cd ~
git clone https://gitlab.com/qemu-project/qemu.git qemu-git
cd qemu-git
git checkout v9.0.0

# 编译（静态安装到 /opt/qemu）
./configure --prefix=/opt/qemu --enable-kvm --enable-sdl \
  --enable-virtfs --target-list=x86_64-softmmu \
  --disable-werror --enable-debug

make -j$(nproc)
sudo make install
```

### 5.3 验证安装

```bash
/opt/qemu/bin/qemu-system-x86_64 --version
# 预期输出：QEMU emulator version 9.0.0

# 添加到 PATH（写入 ~/.bashrc）
echo 'export PATH="/opt/qemu/bin:$PATH"' >> ~/.bashrc
source ~/.bashrc
```

## 6. 内核源码配置

### 6.1 安装内核源码

```bash
# 查看当前内核版本
uname -r
# 预期：6.17.0-20-generic

# 安装源码（linux-source 不是必须的，有 headers 就够编译模块）
sudo apt install -y linux-headers-$(uname -r) linux-source-6.17.0

# 链接源码（驱动开发用）
ls /usr/src/linux-headers-$(uname -r)/
# 预期看到 Makefile、include/、drivers/ 等
```

### 6.2 验证 PCI 相关配置

```bash
# 检查 PCI 配置开启
grep -E "CONFIG_PCI|CONFIG_HAVE_PCI" /boot/config-$(uname -r) | head -20
```

预期关键配置（均为 =y 或 =m）：

```bash
CONFIG_PCI=y
CONFIG_PCI_MSI=y
CONFIG_PCIEPORTBUS=y
CONFIG_PCIEAER=y
CONFIG_PCI_ASPM=y
CONFIG_HOTPLUG_PCI=y
```

## 8. 验证 QEMU PCIe 环境

### 8.1 快速验证 QEMU PCIe RC 正常

```bash
# 检查 QEMU 是否能创建 PCIe ECAM 窗口
/opt/qemu/bin/qemu-system-x86_64 \
  -M virt \
  -m 256M \
  -nographic \
  -enable-kvm \
  -display none \
  -device pci-host-bridge,bus=pcie.0,addr=00.0 \
  -device nec-usb-xhci,addr=1f.0 \
  2>&1 | grep -i "pcie\|ecam\|msi" | head -10
```

### 8.2 在宿主机验证 PCI 总线（不需要进 QEMU）

```bash
# 在宿主机查看 PCI 总线和 PCIe 设备
lspci -nn
# 预期：看到 RC（Host bridge）、各种 PCI 设备

# 查看 PCI 树形结构
lspci -t
# 预期：以树形显示总线-设备-功能关系

# 查看 PCIe 链路状态
lspci -vv -nn -xxx
```

### 8.3 使用 QEMU + GDB 调试内核模块（后续实验基础）

```bash
# 启动 QEMU，等待 GDB 连接
/opt/qemu/bin/qemu-system-x86_64 \
  -M virt \
  -m 512M \
  -kernel /boot/vmlinuz-$(uname -r) \
  -initrd /boot/initrd.img-$(uname -r) \
  -append "console=ttyS0 root=/dev/sda ro" \
  -drive file=/tmp/virtpcitest.qcow2,format=qcow2 \
  -nographic \
  -enable-kvm \
  -s -S &

# 在另一个终端启动 GDB
gdb /usr/lib/debug/boot/vmlinuz-$(uname -r)
(gdb) target remote localhost:1234
# GDB 连接成功后，可以设置断点
```

## 11. 常见错误

### 错误 1：QEMU 编译缺少依赖

```
ERROR: glib-2.48 gthread-2.0 not found
```

**解决**：
```bash
sudo apt install libglib2.0-dev libpixman-1-dev
```

### 错误 2：内核模块编译失败（Makefile 问题）

```
Makefile:*** missing separator
```

**原因**：Makefile 用空格而不是 Tab。
**解决**：确保 Makefile 中 `recipes` 行以 Tab 开头，不是空格。

### 错误 3：QEMU 找不到 KVM

```
Could not access KVM kernel module: Permission denied
```

**解决**：
```bash
# 检查 kvm 是否存在
ls /dev/kvm
# 将当前用户加入 kvm 组
sudo usermod -aG kvm $USER
# 重新登录或执行 newgrp kvm
```

## 参考答案

**1. 为什么 QEMU 8.2 不能用，必须用 9.0+？哪个 PCIe 特性受影响？**

答案：QEMU 8.2 对 PCIe Alternative Routing-ID Forwarding (ARIF) 实现有 bug。ARIF 允许将一部分 Posted 请求重新路由为 Non-Posted，涉及 RCRB（Root Complex Register Block）结构。9.0 修复了这个 bug，否则在虚拟环境下无法正确模拟多 endpoint 场景下的路由行为。

**2. CONFIG_PCI=m 和 CONFIG_PCI=y 有什么区别？驱动模块放在哪里？**

答案：CONFIG_PCI=y 表示 PCI 子系统编译进内核镜像（静态），系统启动即生效。CONFIG_PCI=m 表示编译为模块（ko 文件），可以在系统运行后用 modprobe 加载/卸载。模块路径：`/lib/modules/$(uname -r)/kernel/drivers/pci/`。

**3. ECAM 是什么？QEMU virt 如何模拟 ECAM 空间？**

答案：ECAM（Enhanced Configuration Access Mechanism）将 PCIe 配置空间映射到内存地址空间，CPU 通过内存读写访问配置寄存器。QEMU virt 默认将 ECAM 窗口映射到宿主机的 0xef800000（256MB），Guest OS 通过 ACPI MCFG 表获知这个基址，直接解引用指针访问下游设备的配置空间。

**4. 为什么编译内核模块需要 `uname -r` 对应的 linux-headers，而不是 linux-source？**

答案：linux-headers 只包含编译驱动所需的 Makefile、头文件（include/）、Kconfig，不包含完整源码树。linux-source 是完整内核源码（几百MB）。驱动编译只需要知道"内核导出的 API 长什么样"，不需要内核自身的实现代码，所以 headers 足够且更小。

**5. MSI 和 INTx 中断在 QEMU 中分别如何验证？**

答案：INTx：通过 `lspci -vv` 看 "Interrupt Pin"（INTx#A/B/C/D）和 `/proc/interrupts` 中的中断计数。MSI：在 Guest 内执行 `cat /proc/interrupts | grep MSI`，看 MSI 向量数；用 `setpci -s XX:XX.X MSI_CAP_HDR` 读 MSI Capability 确认地址/数据格式。QEMU 中 MSI 通过写模拟的 RC MSI 地址端口触发软中断。
