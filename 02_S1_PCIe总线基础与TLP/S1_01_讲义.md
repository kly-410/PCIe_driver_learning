# S1_01 — PCIe 总线基础与 TLP

## 本节目标

- 理解 PCIe 总线拓扑：RC / EP / Switch / bridge 的角色分工
- 掌握 TLP（Transaction Layer Packet）的类型与用途
- 理解三种 TLP 路由方式：地址路由、ID 路由、隐式路由

## 前置依赖

无。

---

## A. 原理（是什么）

### PCIe 总线拓扑

PCIe 用点对点交换结构，替代 PCI 的共享总线：

| 组件 | 角色 | 内核对应 |
|------|------|---------|
| **Root Complex (RC)** | CPU 与 PCIe 总线的接口，发起所有事务 | `struct pci_host_bridge` + RC 驱动 |
| **Endpoint (EP)** | 事务的发起者或响应者，终端设备 | `struct pci_dev` |
| **Switch** | 扩展端口，路由 TLP | `drivers/pci/switch/` + 多 `struct pci_dev` |
| **PCIe-to-PCI Bridge** | 兼容老 PCI 设备 | `drivers/pci/host/pci-*.c` |

### TLP 类型

| TLP 类型 | Posted/Non-Posted | 说明 |
|----------|------------------|------|
| **MRd**（Memory Read）| Non-Posted | RC 读 EP BAR，EP 返回 CplD |
| **MWr**（Memory Write）| Posted | RC 写 EP BAR，不等确认 |
| **CfgRd**（Config Read）| Non-Posted | RC 读 EP 配置空间，ID 路由 |
| **CfgWr**（Config Write）| Non-Posted | RC 写 EP 配置空间，ID 路由 |
| **CplD**（Completion with Data）| — | EP 返回 MRd 数据 |
| **Cpl**（Completion without Data）| — | EP 确认 CfgWr 完成 |
| **Msg/MsgD**（Message）| Posted | 带内中断、错误报告、电源管理 |

### 三种路由方式

- **地址路由**：TLP Header 含目标 PCIe 地址，Switch/RC 查路由表决定转发方向。用于 MRd/MWr/IORd/IOWr。
- **ID 路由**：TLP Header 含 Bus/Dev/Func，Switch 查 Bus 号范围决定转发方向。用于 CfgRd/CfgWr/Cpl/CplD。
- **隐式路由**：路由器根据端口状态决定方向，不看地址也不看 ID。用于 Message（Msg/MsgD）。

---

## B. 为什么这样设计

### 为什么区分 Posted 和 Non-Posted？

Posted 写不阻塞 RC，可流水线发送提高性能。Non-Posted 读必须等数据返回——RC 不知道 EP 存了什么，必须等 EP 主动返回。

### 为什么用交换结构而非共享总线？

PCI 所有设备争用同一总线带宽，扩展性受总线频率限制。PCIe 点对点链路独立，每个设备独享带宽，支持多链路并行（x1/x4/x8/x16），扩展性更强。

### 为什么要用隐式路由？Msg/MsgD 为什么不用 ID 路由？

**PCIe 初始化三阶段**：

```
1. 硬件复位（Reset）
   → Link Training（链路训练）正在进行
   → LTSSM 状态机在跑
   → 此时设备还没有 Bus/Dev/Func 地址（没有 ID）

2. BIOS/UEFI 枚举
   → OS 开始扫 PCI 总线
   → 给每个设备分配 Bus 号（分配 ID）
   → 这时候才知道每个设备的 Bus/Dev/Func

3. 驱动加载
   → 驱动匹配，配置 BAR，开始工作
```

**错误消息（ERR_COR / ERR_NONFATAL / ERR_FATAL）什么时候发？**

链路协商失败、CRC 错误、字符丢失——这些错误**可能在阶段 1（链路训练阶段）就发生了**。此时设备还没有 Bus 号，根本不知道自己的 ID 是多少。

如果 ERR 消息用 ID 路由：需要 TLP Header 里写 Requester ID（Bus/Dev/Func）→ 但阶段 1 根本没有 ID → ERR 消息发不出去 → RC 不知道链路出错了。

**隐式路由的解决方案**：Msg Header 里有一个 "Routing Field"，只有 3 种情况：

```
1. Route to RC
   → 消息往上走，终止于 RC（EP 发 ERR_* 通知 RC 出了错）

2. Broadcast from RC
   → 消息从 RC 往下游广播，所有 EP 都能收到（RC 发 PME_Turn_Off，通知所有人断电）

3. Local / Passthrough
   → 进了哪端口就在哪终止，不转发（特殊消息在 Switch 内部处理）
```

设备只需要知道"把消息往上游端口送"就行，不需要知道 RC 的 ID，也不需要知道自己的 ID。

**总结**：隐式路由是 PCIe 里"不需要知道对方是谁，直接往上送或往下广播"的机制，专门给带内消息用的——因为这些消息（错误通知、电源管理、中断）在设备还没有 ID 的时候就可能需要发送。

## C. 常见错误案例

### 错误 1：MRd 没有收到 CplD（地址路由配置错误）

**现象**：RC 发 MRd，EP 无响应，返回全 F。

**排查**：检查 ATU Outbound 窗口基址是否和 EP BAR 匹配。

### 错误 2：MSI 中断不触发（MsgD 路由错误）

**现象**：EP 驱动写 MSI 地址，RC 未收到中断。

**排查**：确认 RC 分配的 MSI 地址/数据字段和 EP MSI Cap 一致。

---

## D. 决策树

```
发起配置访问 → CfgRd/CfgWr → ID路由（Bus/Dev/Func）
发起内存访问 → MRd/MWr → 地址路由（目标PCIe地址）
发起完成响应 → CplD/Cpl → ID路由（Requester ID）
发起消息    → Msg/MsgD → 隐式路由（根据端口状态决定）
```

## 源码速查（内核视角）

> S1_01 主要是 PCIe 协议规范，内核源码不直接参与 TLP 发送/接收。以下是内核中用于描述 PCIe 设备能力的结构体定义。

| 源码文件 | 关键内容 | 行号 |
|---------|---------|------|
| `include/uapi/linux/pci_regs.h` | `PCI_EXP_FLAGS`（版本/端口类型）| 483 |
| `include/uapi/linux/pci_regs.h` | `PCI_EXP_LNKSTA`（当前链路速率/宽度）| 577 |
| `include/uapi/linux/pci_regs.h` | `PCI_EXP_LNKCTL`（链路控制/ASPM）| 564 |
| `include/uapi/linux/pci_regs.h` | `PCI_EXP_DEVSTA`（设备状态/错误检测）| 537 |
| `include/linux/pci.h` | `struct pci_dev` 中的 `pcie_cap`（缓存 PCIe Cap 偏移）| ~380 |
| `drivers/pci/pci.c` | `pcie_get_readrq/pcie_set_readrq`（读请求大小/MRS）| ~2620 |

**关键 PCIe 寄存器（从 RC/EP 配置空间读取）**：

```c
/* PCIe Capability ID = 0x10，位于 Capability Pointer (0x34) */
`#define PCI_EXP_FLAGS     0x02  // PCIe Cap Header（版本 + 端口类型）`
`#define PCI_EXP_LNKSTA   0x12  // Link Status（当前协商速率和宽度）`
`#define PCI_EXP_LNKCTL   0x10  // Link Control（ASPM/时钟管理）`
`#define PCI_EXP_DEVSTA   0x0a  // Device Status（错误状态 + Transactions Pending）`

/* Link Status 位定义（PCI_EXP_LNKSTA）*/
`#define PCI_EXP_LNKSTA_CLS 0x000f   // 当前链路速率（Gen1/2/3/4/5）`
`#define PCI_EXP_LNKSTA_CLS_2_5  0x01 // Gen1 (2.5 GT/s)`
`#define PCI_EXP_LNKSTA_CLS_5_0  0x02 // Gen2 (5.0 GT/s)`
`#define PCI_EXP_LNKSTA_CLS_8_0  0x03 // Gen3 (8.0 GT/s)`
`#define PCI_EXP_LNKSTA_NLW     0x03f0 // 协商宽度（x1/x2/x4/x8/x16）`
```
