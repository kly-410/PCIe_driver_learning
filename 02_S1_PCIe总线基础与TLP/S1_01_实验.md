## 实验 1：PCIe 总线扫描

**目标**：用 scan 工具扫描宿主机 PCIe 总线，理解拓扑结构。

**步骤**：
```bash
### 编译
make scan

### 扫描所有 PCIe 设备
sudo ./scan

### 对比 lspci
lspci -nn

### 过滤 PCIe 设备
lspci -nn | grep PCIe
```

**预期结果**：00:00.0 是 RC（Host Bridge），其他是 EP 或 Switch 端口。

**验收标准**：
- [ ] scan 工具正常运行
- [ ] 能区分 Type 0（EP）和 Type 1（RC/Bridge）

---

## 实验 2：TLP 类型识别

**目标**：用 tlp_type 工具识别 RC 和 EP 的 Device Type，理解 TLP 角色。

**步骤**：
```bash
### 编译
make tlp_type

### 分析 RC（00:00.0）
sudo ./tlp_type 00:00.0

### 分析第一个 EP（通常 01:00.0 virtio-net 或 nvme）
sudo ./tlp_type 01:00.0

### 列出所有 PCIe 设备
lspci -nn | awk '{print $1}' | while read dev; do
    echo "=== $dev ===" && sudo ./tlp_type $dev 2>/dev/null || true
done
```

**验收标准**：
- [ ] RC 的 Device Type = Root Port
- [ ] EP 的 Device Type = PCIe Endpoint
- [ ] 能解释 RC 发给 EP 的 TLP 类型

## 实验 3：链路状态读取

**目标**：读取链路协商速率和宽度，理解 Gen/Width 协商过程。

**步骤**：
```bash
### 编译
make link_status

### 读取 RC 链路状态
sudo ./link_status 00:00.0

### 读取 EP 链路状态
sudo ./link_status 01:00.0

### 用 lspci 交叉验证
lspci -vv -nn -s 00:00.0 | grep -i "lnksta\|speed\|width"
```

**验收标准**：
- [ ] 能读出 Gen1/2/3/4/5 速率
- [ ] 能读出 x1/x2/x4/x8/x16 宽度
- [ ] 理解协商宽度受链路质量限制

## 实验 4：TLP 路由验证（可选）

**目标**：用 setpci 触发配置访问，观察 CfgRd/CfgWr 的 ID 路由行为。

**步骤**：
```bash
### 读取 RC 的 Vendor/Device ID（CfgRd → ID 路由）
sudo ./tlp_type 00:00.0 | grep "Vendor ID"

### 对比：直接用 setpci 读（CfgRd）
sudo setpci -s 00:00.0 0.w

### 读 EP 的 BAR0（通过配置空间）
sudo setpci -s 01:00.0 10.l

### 读 MSI Cap Message Data（EP 发 MSI 时写入的数据）
sudo ./tlp_type 01:00.0 | grep "MSI"
```

**验收标准**：
- [ ] 能用 setpci 读取配置寄存器
- [ ] 理解 ID 路由不依赖地址，只依赖 Bus/Dev/Func

### 改错题（Challenge）

代码中故意埋了一个 bug，运行或审查后找出问题。

**实验代码**（`S1_01_tlp_type.c`，在 `02_S1_PCIe总线基础与TLP/` 目录下）：

下面的简化代码用来发送一个 PCIe Memory Write TLP，写入 EP 的 BAR0 空间：

```c
/* S1_01_tlp_type.c 片段 —— 故意埋了一个 TLP Type bug */
#include <stdio.h>
#include <pci/pci.h>

void write_bar(struct pci_dev *dev, int bar, u32 value)
{
    /* 计算 ECAM 地址 */
    u64 ecam = 0xef800000;
    ecam += (dev->bus->number  << 20);
    ecam += (dev->devfn >> 3   << 15);  /* Dev */
    ecam += (dev->devfn & 0x7  << 12);  /* Func */
    ecam += bar;

    /* 构造 TLP Header：Fmt=2DW，Type 字段 */
    u32 tlp_header[2] = {
        0x40000000,        /* Fmt=2DW(0x40) + Type=0x00(MRd) */
        0xfee00000         /* EP BAR0 地址（目标地址）*/
    };
    tlp_header[0] |= (value & 0xFFFFFFF); /* 低 27 位放数据 */

    /* 写入 RC 的 outbound 窗口寄存器，触发 TLP 发送 */
    *(volatile u32 *)ecam = tlp_header[0];
    *(volatile u32 *)(ecam + 4) = tlp_header[1];
}

int main(int argc, char **argv)
{
    struct pci_dev *dev = pci_get_dev(0, 0, atoi(argv[2]), atoi(argv[3]));
    /* 写入 BAR0，值 = 0x12345678 */
    write_bar(dev, 0x10, 0x12345678);  /* BAR0 偏移 = 0x10 */
    return 0;
}
```

**问题**：上述代码希望发送一个 **Memory Write（MWr）**，但 TLP Header 的 Fmt+Type 编码写成了 MRd（Memory Read）。

1. 找到 Fmt/Type 编码错误的那一行
2. 写出正确的 Fmt+Type 组合

> 提示：PCIe TLP 中 MRd 的 Type 字段为 `0x00`，而 MWr 的 Type 字段为 `0x01`（PCIe 5.0 Spec Table 8-1）。Fmt 字段两者在普通 Memory Request 中都是 `0x40`（2DW）。

预期答案：`0x40000000` 应改为 `0x40000001`（Fmt=2DW(0x40) + Type=MWr(0x01)），当前 Type=0x00 表示 MRd，导致数据被读走而不是写入 EP。
