# S1_01 — PCIe 总线基础与 TLP

## 本节目标

- 理解 PCIe 总线拓扑：RC / EP / Switch / bridge 的角色分工
- 掌握 TLP（Transaction Layer Packet）的类型与用途
- 理解三种 TLP 路由方式：地址路由、ID 路由、隐式路由

## 前置依赖

无。

---

## A. 原理（是什么）

### PCIe 总线拓扑

PCIe 用点对点交换结构，替代 PCI 的共享总线：

| 组件 | 角色 | 内核对应 |
|------|------|---------|
| **Root Complex (RC)** | CPU 与 PCIe 总线的接口，发起所有事务 | `struct pci_host_bridge` + RC 驱动 |
| **Endpoint (EP)** | 事务的发起者或响应者，终端设备 | `struct pci_dev` |
| **Switch** | 扩展端口，路由 TLP | `drivers/pci/switch/` + 多 `struct pci_dev` |
| **PCIe-to-PCI Bridge** | 兼容老 PCI 设备 | `drivers/pci/host/pci-*.c` |

### TLP 类型

| TLP 类型 | Posted/Non-Posted | 说明 |
|----------|------------------|------|
| **MRd**（Memory Read）| Non-Posted | RC 读 EP BAR，EP 返回 CplD |
| **MWr**（Memory Write）| Posted | RC 写 EP BAR，不等确认 |
| **CfgRd**（Config Read）| Non-Posted | RC 读 EP 配置空间，ID 路由 |
| **CfgWr**（Config Write）| Non-Posted | RC 写 EP 配置空间，ID 路由 |
| **CplD**（Completion with Data）| — | EP 返回 MRd 数据 |
| **Cpl**（Completion without Data）| — | EP 确认 CfgWr 完成 |
| **Msg/MsgD**（Message）| Posted | 带内中断、错误报告、电源管理 |

### 三种路由方式

- **地址路由**：TLP Header 含目标 PCIe 地址，Switch/RC 查路由表决定转发方向。用于 MRd/MWr/IORd/IOWr。
- **ID 路由**：TLP Header 含 Bus/Dev/Func，Switch 查 Bus 号范围决定转发方向。用于 CfgRd/CfgWr/Cpl/CplD。
- **隐式路由**：路由器根据端口状态决定方向，不看地址也不看 ID。用于 Message（MsgD）。

---

## B. 为什么这样设计

### 为什么区分 Posted 和 Non-Posted？

Posted 写不阻塞 RC，可流水线发送提高性能。Non-Posted 读必须等数据返回——RC 不知道 EP 存了什么，必须等 EP 主动返回。

### 为什么用交换结构而非共享总线？

PCI 所有设备争用同一总线带宽，扩展性受总线频率限制。PCIe 点对点链路独立，每个设备独享带宽，支持多链路并行（x1/x4/x8/x16），扩展性更强。

## C. 常见错误案例

### 错误 1：MRd 没有收到 CplD（地址路由配置错误）

**现象**：RC 发 MRd，EP 无响应，返回全 F。

**排查**：检查 ATU Outbound 窗口基址是否和 EP BAR 匹配。

### 错误 2：MSI 中断不触发（MsgD 路由错误）

**现象**：EP 驱动写 MSI 地址，RC 未收到中断。

**排查**：确认 RC 分配的 MSI 地址/数据字段和 EP MSI Cap 一致。

---

## D. 决策树

```
发起配置访问 → CfgRd/CfgWr → ID路由（Bus/Dev/Func）
发起内存访问 → MRd/MWr → 地址路由（目标PCIe地址）
发起完成响应 → CplD/Cpl → ID路由（Requester ID）
发起消息    → MsgD → 隐式路由（根据端口状态决定）
```
