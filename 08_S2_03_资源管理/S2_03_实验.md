### 改错题（Challenge）

代码中故意埋了一个 bug，运行或审查后找出问题。

**实验代码**（`S2_03_resource.c`，在 `08_S2_03_资源管理/` 目录下）：

```c
/* S2_03_resource.c —— 故意埋了 ioremap/iounmap 不配对 bug */
#include <linux/pci.h>
#include <linux/module.h>

static struct pci_dev *dev;
static void __iomem *bar0_virt;

static int my_probe(struct pci_dev *pdev, const struct pci_device_id *id)
{
    u32 val;

    dev = pdev;

    /* 映射 BAR0 */
    bar0_virt = ioremap(pci_resource_start(dev, 0), 0x1000);
    if (!bar0_virt) {
        dev_err(&dev->dev, "ioremap failed\n");
        return -ENOMEM;
    }

    /* 读取 BAR0 第一个寄存器 */
    val = ioread32(bar0_virt);
    dev_info(&dev->dev, "BAR0[0x00] = 0x%08x\n", val);

    /* 函数结束，没有调用 iounmap(bar0_virt)！*/
    return 0;
}

static void my_remove(struct pci_dev *pdev)
{
    /* remove 时也没有 iounmap */
    dev_info(&dev->dev, "remove called, leaking bar0_virt\n");
}

static struct pci_driver my_driver = {
    .name = "s2_03_resource",
    .probe = my_probe,
    .remove = my_remove,
};

module_pci_driver(my_driver);
MODULE_LICENSE("GPL");
```

**问题**：`bar0_virt` 在 probe 时 `ioremap`，但既没有在函数返回前 `iounmap`，也没有在 remove 时 `iounmap`。这会导致：

1. 内核虚拟地址空间泄漏（ioremap 占用 4KB/页）
2. 驱动卸载后，映射仍然存在，后续其他驱动或内核代码可能误用这个已卸载驱动的映射地址

1. 找到缺少资源释放的位置（两处）
2. 写出正确的修复方式

> 提示：`ioremap()` 配对的释放函数是 `iounmap()`。典型模式：在 probe 失败路径和 remove 函数中都要调用 `iounmap(bar0_virt)`。

预期答案：有两处缺少 iounmap：
1. `my_probe` 成功映射后，如果后续逻辑返回错误（如读取失败），应立即 iounmap
2. `my_remove` 必须调用 `iounmap(bar0_virt)` 释放映射，否则 bar0_virt 永久泄漏，导致内核 vmalloc 区域逐渐耗尽
