# S1_04 — BAR 地址映射与解码

## 本节目标

- 理解 BAR（Base Address Register）的作用：申请 I/O 或 Memory 地址空间
- 掌握 BAR 类型判断：IO vs MEM，32-bit vs 64-bit，PREFETCH
- 掌握 pci_request_regions() / pci_iomap() / ioremap() 的使用
- 理解内存屏障（rmb/wmb/mb）在 MMIO 中的作用

## 前置依赖

S1_01（TLP 类型）、S1_03（配置空间）。

---

## A. 原理（是什么）

### BAR 的本质

BAR 是 EP 配置空间中的寄存器（Offset 0x010~0x027，共 6 个）。

软件通过 BAR **告诉 EP**："你的寄存器映射到哪个 PCIe 地址，这样 RC 才能访问你。"

解析过程：
1. 向 BAR 写入 0xFFFFFFFF
2. 读回，得到解码后的 size 字段
3. 系统分配地址后写入 BAR

### BAR 类型判断

读 BAR 的 bit 0：
- **bit 0 = 1**：I/O Space（x86 专用）
- **bit 0 = 0**：Memory Space，bit[2:1] 决定类型

| BAR[bit 2:1] | 类型 | 说明 |
|--------------|------|------|
| 00 | 32-bit MEM | 普通内存，不预取 |
| 01 | 64-bit MEM | 低 32 位，bit 1=01 表示高位在下一个 BAR |
| 10 | 32-bit MEM | 预取（可写合并）|
| 11 | 保留 | — |

bit 3（PREFETCH）= 1：可预取，允许写合并，读不会改变 EP 状态。

### MMIO 地址映射

CPU 只能通过虚拟地址访问内存。EP BAR 地址是 PCIe 物理地址，需要通过 ioremap() 建立页表映射：

```
BAR 地址（PCIe 物理地址）
    ↓  ioremap()
CPU 虚拟地址（__iomem 指针）
    ↓  ioread32/iowrite32
PCIe 链路（TLP）
    ↓
EP 寄存器
```

### 内存屏障

PCIe 写操作可能是 Posted（MWr），不保证顺序。屏障保证读写顺序：

| 屏障 | 作用 |
|------|------|
| `rmb()` | 读屏障：之后的读不会被 CPU 重排到之前 |
| `wmb()` | 写屏障：之后的写不会被 CPU 重排到之前 |
| `mb()` | 读写全屏障 |

## B. 为什么这样设计

### 为什么要用 readl/writel，不能直接解引用指针？

三个原因：① 编译器优化——直接解引用可能因"无可见副作用"被优化掉或重排序；② Posted Write——x86 store 指令可能 Posted（不等待刷新到 cache），需要 wmb() 强制刷写缓冲区；③ ioread32/iowrite32 是平台相关的 MMIO 访问函数，确保 Posted Write 被正确发出。

### 为什么 wmb() 在 DMA 描述符下发中必须？

典型错误场景：
```c
iowrite32(data_addr, bar + DMA_ADDR_REG);  // 写描述符地址
iowrite32(START_CMD, bar + DMA_CTRL_REG);   // 启动命令 ←Posted，可能先到！
```
如果没有 wmb()，START_CMD 可能先被 EP 看到（EP 先收到"启动"但还不知道数据地址）。正确写法：
```c
iowrite32(data_addr, bar + DMA_ADDR_REG);
wmb();  // 保证 addr 在 cmd 之前被 EP 看到
iowrite32(START_CMD, bar + DMA_CTRL_REG);
```

### 为什么要 pci_request_regions？

系统中可能有多个驱动。如果两个驱动同时 ioremap 同一个 BAR，会冲突。pci_request_regions() 向系统注册 BAR 所有权，防止重复申请。

## C. 常见错误案例

### 错误 1：ioremap 成功，但读写全 F

**现象**：ioremap() 返回非 NULL，但 readl() 返回 0xFFFFFFFF。

**排查**：检查 Command Register 的 Memory Space Enable 位（bit 1）是否为 1。

### 错误 2：64-bit BAR 申请失败

**现象**：BAR0 是 64-bit，但 BAR1 被另一个功能占用。

**原因**：64-bit BAR 必须连续（BAR0+BAR1），不能跳过 BAR1。

### 错误 3：DMA 描述符顺序错误（无 wmb）

**现象**：EP 先收到启动命令，后收到数据地址，DMA 传输错误数据。

**解决**：在描述符写入后、启动命令前加 wmb()。

## D. 决策树

```
配置 EP BAR 空间
├── BAR[bit 0] = 1
│   └── I/O BAR → 用 inb/outb（x86 专用）
├── BAR[bit 0] = 0 且 BAR[bit 2:1] = 01
│   └── 64-bit MEM BAR → BAR0 存低位，BAR1 存高位
└── BAR[bit 0] = 0 且 BAR[bit 2:1] = 00/10
    └── 32-bit MEM BAR → 直接解析

访问 BAR 空间
├── pci_request_regions() → 向系统注册所有权
├── pci_iomap() 或 ioremap() → 建立虚拟地址映射
└── ioread32/iowrite32 → MMIO 读写，禁止直接解引用

写寄存器后加屏障
└── wmb() → 确保 Posted 写顺序
```
