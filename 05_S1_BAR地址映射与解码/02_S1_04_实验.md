# S1_04_实验 — BAR 地址映射与解码

> 前提：安装 libpci-dev
> `sudo apt install libpci-dev`

---

## 实验 1：BAR 大小解析

**目标**：解析 BAR 类型和申请的空间大小，理解"写入全 1 读回"法。

**步骤**：
```bash
# 编译
make bar_decode

# 解析 EP 的 BAR
sudo ./bar_decode 01:00.0

# 解析 NVMe（如果有）
lspci -nn | grep NVMe | awk '{print $1}' | while read dev; do
    echo "=== $dev ===" && sudo ./bar_decode $dev 2>/dev/null
done

# 用 setpci 交叉验证（手动写全 1 读回）
sudo setpci -s 01:00.0 10.l   # 读 BAR0 当前值
# 手动测试：
# setpci -s 01:00.0 10.l=FFFFFFFF
# setpci -s 01:00.0 10.l       # 读回，得到 size 编码
```

**预期结果**：EP 的 BAR 通常是 64-bit MEM（ virtio-net 申请 4KB，nvme 申请 4KB 或更大）。

**验收标准**：
- [ ] BAR_decode 正常运行
- [ ] 能区分 IO BAR 和 MEM BAR
- [ ] 能识别 64-bit BAR

## 实验 2：MMIO 屏障演示

**目标**：运行 mmio 程序，理解屏障的作用。

**步骤**：
```bash
# 编译（不需要 libpci）
make mmio

# 运行（不需要 root，不需要硬件）
./mmio
```

**预期结果**：程序打印屏障决策树和 DMA 描述符下发示例。

**验收标准**：
- [ ] 能解释为什么 wmb() 在 DMA 描述符下发中必须
- [ ] 能解释 Posted MWr 和屏障的关系

## 实验 3：BAR 资源管理

**目标**：读取 pci_dev->resource[]，理解 BAR 在系统内存中的映射。

**步骤**：
```bash
# 编译
make resource

# 读取 RC 的 BAR
sudo ./resource 00:00.0

# 读取 EP 的 BAR
sudo ./resource 01:00.0

# 对比 /proc/iomem
cat /proc/iomem | grep -i "pci\|0001" | head -20
```

**验收标准**：
- [ ] resource 程序正常运行
- [ ] 能解释 pci_request_regions 的作用（防止 BAR 冲突）
- [ ] 能读取 /proc/iomem 找到 BAR 对应的系统地址

## 实验 4：ioregion 冲突模拟（可选）

**目标**：理解为什么需要 pci_request_regions。

**步骤**：
```bash
# 查看当前系统的 BAR 分配
cat /sys/bus/pci/devices/*/resource | head -20

# 查找被哪个驱动占用的 BAR
for f in /sys/bus/pci/devices/*/driver; do
    echo "$f: $(cat $f 2>/dev/null | xargs basename 2>/dev/null)"
done | grep -v ": $"

# 查看特定设备的 resource 路径
ls -la /sys/bus/pci/devices/0000:01:00.0/resource*
```

**验收标准**：
- [ ] 理解 /proc/iomem 中的 BAR 地址是"系统分配的 CPU 物理地址"
- [ ] 理解 pci_request_regions 防止多个驱动同时 ioremap 同一个 BAR
