# S1_04_实验 — BAR 地址映射与解码

> 前提：安装 libpci-dev
> `sudo apt install libpci-dev`

---

## 实验 1：BAR 大小解析

**目标**：解析 BAR 类型和申请的空间大小，理解"写入全 1 读回"法。

**步骤**：
```bash
### 编译
make bar_decode

### 解析 EP 的 BAR
sudo ./bar_decode 01:00.0

### 解析 NVMe（如果有）
lspci -nn | grep NVMe | awk '{print $1}' | while read dev; do
    echo "=== $dev ===" && sudo ./bar_decode $dev 2>/dev/null
done

### 用 setpci 交叉验证（手动写全 1 读回）
sudo setpci -s 01:00.0 10.l   # 读 BAR0 当前值
### 手动测试：
#### setpci -s 01:00.0 10.l=FFFFFFFF
#### setpci -s 01:00.0 10.l       # 读回，得到 size 编码
```

**预期结果**：EP 的 BAR 通常是 64-bit MEM（ virtio-net 申请 4KB，nvme 申请 4KB 或更大）。

**验收标准**：
- [ ] BAR_decode 正常运行
- [ ] 能区分 IO BAR 和 MEM BAR
- [ ] 能识别 64-bit BAR

sudo ./resource 00:00.0

## 实验 4：ioregion 冲突模拟（可选）

**目标**：理解为什么需要 pci_request_regions。

**步骤**：
```bash
### 查看当前系统的 BAR 分配
cat /sys/bus/pci/devices/*/resource | head -20

### 查找被哪个驱动占用的 BAR
for f in /sys/bus/pci/devices/*/driver; do
    echo "$f: $(cat $f 2>/dev/null | xargs basename 2>/dev/null)"
done | grep -v ": $"

### 查看特定设备的 resource 路径
ls -la /sys/bus/pci/devices/0000:01:00.0/resource*
```

**验收标准**：
- [ ] 理解 /proc/iomem 中的 BAR 地址是"系统分配的 CPU 物理地址"

### 改错题（Challenge）

代码中故意埋了一个 bug，运行或审查后找出问题。

**实验代码**（`S1_04_mmio.c`，在 `05_S1_BAR地址映射与解码/` 目录下）：

下面的代码向 EP 的 DMA 控制寄存器写入描述符地址，然后发送启动命令：

```c
/* S1_04_mmio.c 片段 —— 故意埋了 wmb() 缺失 bug */
#include <stdio.h>
#include <stdint.h>
#include <linux/pci.h>

/* 假设 BAR0 已 ioremap 到 bar0_virt */
volatile void *bar0_virt;

void dma_start(void *data_buf, size_t size)
{
    uintptr_t data_addr = (uintptr_t)data_buf;

    /* 写 DMA 描述符地址寄存器 */
    iowrite32(lower_32_bits(data_addr), bar0_virt + 0x00);
    iowrite32(upper_32_bits(data_addr), bar0_virt + 0x04);

    /* 写 DMA 描述符长度寄存器 */
    iowrite32(size, bar0_virt + 0x08);

    /* 启动 DMA 传输 ← 没有 wmb()！ */
    iowrite32(0x00000001, bar0_virt + 0x10);  /* START bit = 1 */
}

int main(void)
{
    char buf[64];
    /* 填充数据 */
    memset(buf, 0xAB, sizeof(buf));

    /* 启动 DMA，把 buf 的内容传给 EP */
    dma_start(buf, sizeof(buf));

    /* 验证：EP 应该已经收到了完整描述符 */
    return 0;
}
```

**问题**：PCIe 写操作是 Posted（不等待对方确认就返回）。EP 内部的 DMA 引擎可能先收到 START 命令，后收到数据地址——DMA 传输错误的数据甚至跑飞。

1. 找到缺少屏障的那一段代码
2. 在启动命令前插入正确的屏障调用，并解释为什么需要它

> 提示：x86 的 store 指令是 weakly ordered，数据先进入 store buffer，不一定立刻到 PCIe 控制器。`wmb()` 强制 flush store buffer，保证 addr/length 在 START 之前到达 EP。

预期答案：在 `iowrite32(0x00000001, bar0_virt + 0x10)` 之前插入 `wmb()`。原因：PCIe Posted Write 不保证顺序，没有 wmb() 时 EP 可能先收到 START 再收到地址，导致 DMA 描述符不完整。
