# S1_04 — BAR 地址映射与解码

## 本节目标

- 理解 BAR（Base Address Register）的作用：申请 I/O 或 Memory 地址空间
- 掌握 BAR 类型判断：IO vs MEM，32-bit vs 64-bit，PREFETCH
- 掌握 pci_request_regions() / pci_iomap() / ioremap() 的使用
- 理解内存屏障（rmb/wmb/mb）在 MMIO 中的作用

## 前置依赖

S1_01（TLP 类型）、S1_03（配置空间）。

---

## A. 原理（是什么）

### BAR 的本质

BAR 是 EP 配置空间中的寄存器（Offset 0x010~0x027，共 6 个）。

软件通过 BAR **告诉 EP**："你的寄存器映射到哪个 PCIe 地址，这样 RC 才能访问你。"

解析过程：
1. 向 BAR 写入 0xFFFFFFFF
2. 读回，得到解码后的 size 字段
3. 系统分配地址后写入 BAR

### BAR 类型判断

读 BAR 的 bit 0：
- **bit 0 = 1**：I/O Space（x86 专用）
- **bit 0 = 0**：Memory Space，bit[3] 和 bit[2] 决定类型

**源码依据**（`include/uapi/linux/pci_regs.h:102-109`）：
```c
`#define PCI_BASE_ADDRESS_SPACE            0x01  // bit0: 0=memory, 1=I/O`
`#define PCI_BASE_ADDRESS_SPACE_IO         0x01`
`#define PCI_BASE_ADDRESS_SPACE_MEMORY     0x00`
`#define PCI_BASE_ADDRESS_MEM_TYPE_MASK    0x06  // bit[2:1] mask`
`#define PCI_BASE_ADDRESS_MEM_TYPE_32      0x00  // 32-bit address`
`#define PCI_BASE_ADDRESS_MEM_TYPE_1M      0x02  // Below 1M [obsolete]`
`#define PCI_BASE_ADDRESS_MEM_TYPE_64      0x04  // 64-bit address（bit2=1）`
`#define PCI_BASE_ADDRESS_MEM_PREFETCH     0x08  // bit3: prefetchable`
```

**正确判断逻辑**：
```
BAR & 0x01 == 1  →  I/O Space
BAR & 0x01 == 0  →  Memory Space

  (BAR & 0x06) == 0x00  →  32-bit MEM（bit2=0）
  (BAR & 0x06) == 0x04  →  64-bit MEM（bit2=1，BAR[n]存低32位，BAR[n+1]存高32位）
  BAR & 0x08 == 0x08  →  可预取（prefetchable）
```

**判断示例**：
```
BAR 值 = 0x00000004  →  Memory Space，type=(4&6)=4 → 64-bit MEM，non-prefetch
BAR 值 = 0x00000008  →  Memory Space，type=(8&6)=0 → 32-bit MEM，prefetch
BAR 值 = 0x0000000C  →  Memory Space，type=(C&6)=4 → 64-bit MEM，prefetch
BAR 值 = 0x00000001  →  I/O Space
```

> **补充自用户提问**：bit 0 = 1 为什么是 x86 专用？
> 因为 I/O Space 是 x86 架构独有的（IA-32 的 IN/OUT 指令），ARM/RISC-V 没有 I/O Space，所有外设都用 Memory Mapped I/O。Linux 内核中 `include/uapi/linux/pci_regs.h:102` 用 `PCI_BASE_ADDRESS_SPACE` 区分这两种地址空间。

### MMIO 地址映射（完整软件路径）

CPU 只能通过虚拟地址访问内存。BAR 地址是 PCIe 物理地址，需要通过 ioremap() 建立页表映射：

```
BAR 地址（PCIe 物理地址，如 0xFEE00000）
    ↓ ioremap()
CPU 虚拟地址（__iomem 指针）
    ↓ iowrite32()
PCIe TLP（Posted Memory Write）
    ↓ PCIe 链路
EP 内部寄存器
```

**各层源码**：

| 步骤 | 源码 |
|:---|:---|
| `ioremap()` 建立映射 | `arch/x86/mm/ioremap.c:343` → `__ioremap_caller()`，建页表返回 `__iomem` 指针 |
| `iowrite32(val, addr)` | `include/asm-generic/io.h:937` → `writel(addr)` |
| `writel()`（x86） | `arch/x86/include/asm/io.h:85` → `__writel()` → 发起 Posted MWr TLP |

> 来源：`arch/x86/mm/ioremap.c:343`
```c
void __iomem *ioremap(resource_size_t phys_addr, unsigned long size)
{
    enum page_cache_mode pcm = _PAGE_CACHE_MODE_UC_MINUS;
    return __ioremap_caller(phys_addr, size, pcm,
                            __builtin_return_address(0), false);
}
```

> 来源：`include/asm-generic/io.h:937`
```c
#define iowrite32 iowrite32
static inline void iowrite32(u32 value, volatile void __iomem *addr)
{
    writel(value, addr);
}
```

> 来源：`arch/x86/include/asm/io.h:85`
```c
#define writel writel
#define writel_relaxed(v, a) __writel(v, a)
#define __raw_writel __writel
```

PCIe TLP 这一步是硬件行为：RC 收到 `writel` 后，通过 PCIe 控制器发起 Posted Memory Write TLP，EP 收到后更新内部寄存器——软件层面管不到这一层。

> **补充自用户提问**：`ioremap` 是内核函数，用户态程序怎么访问外设？
> 用户态用 `mmap()` 将 `/dev/mem` 或 `/dev/pciep_*` 映射到用户虚拟地址。`ioremap` 只在内核驱动里用，因为需要建页表（只有内核有权限调用）。

### 内存屏障

PCIe 写操作可能是 Posted（MWr），不保证顺序。屏障保证读写顺序：

| 屏障 | 作用 |
|:---|:---|
| `rmb()` | 读屏障：之后的读不会被 CPU 重排到之前 |
| `wmb()` | 写屏障：之后的写不会被 CPU 重排到之前 |
| `mb()` | 读写全屏障 |

## B. 为什么这样设计

### 为什么要用 ioread32/iowrite32，不能直接 `*ptr = val`？

直接解引用指针在 DRAM 场景没问题，但 BAR 地址是外设寄存器，行为完全不同：

```c
// 错误：直接解引用
volatile u32 *reg = (u32 *)0xFEE00000;
*reg = 0x1234;  // 编译器可能优化掉（无可见副作用）

// 正确：用 iowrite32
iowrite32(0x1234, bar + 0x10);  // 明确告知 CPU"这是外设寄存器"
```

**三个原因**：

1. **编译器优化**：直接解引用可能因"无可见副作用"被优化掉或重排序，`volatile` 只解决部分问题
2. **平台相关**：`ioremap` 返回的地址在不同架构（x86/ARM/RISC-V）上映射方式不同，`iowrite32` 是平台适配层
3. **Posted Write**：x86 store 指令是 weakly ordered，`iowrite32` 确保 Posted MWr 被正确发出

> 来源：`include/asm-generic/io.h:937`
```c
static inline void iowrite32(u32 value, volatile void __iomem *addr)
{
    writel(value, addr);
}
```

### 为什么要 pci_request_regions？

系统中可能有多个驱动同时访问同一个 BAR：

```
驱动A：想用 BAR0，觉得 BAR0 是自己的
驱动B：也用了 BAR0，觉得 BAR0 也是自己的
→ 两个驱动同时 ioremap 同一个地址 → 地址空间冲突
```

`pci_request_regions()` 向系统注册 BAR 所有权，防止重复申请。系统中只有第一个申请者能成功拿到 BAR，后续驱动申请会返回 -EBUSY。

> 来源：`drivers/pci/pci_request_regions`（kernel 源码）
```
pci_request_regions() → 检查 res->flags → 调用 request_resource()
```

### 为什么 wmb() 在 DMA 描述符下发中必须？

典型错误场景：
```c
iowrite32(data_addr, bar + DMA_ADDR_REG);  // 写描述符地址
iowrite32(START_CMD, bar + DMA_CTRL_REG);   // 启动命令 ←Posted，可能先到！
```

PCIe 写是 Posted（不等待对方确认就返回）。EP 可能先收到 START，再收到地址——EP 收到"启动"时还不知道数据在哪，DMA 跑飞。

**为什么 Posted 写会乱序？**

x86 的 store 指令是 weakly ordered，数据先进入 store buffer，不一定立刻到 PCIe 控制器。CPU 认为"写完成"了，但 EP 还没收到。

```c
// 正确写法：
iowrite32(data_addr, bar + DMA_ADDR_REG);
wmb();  // 强制 flush store buffer，保证 addr 在 cmd 之前被 EP 看到
iowrite32(START_CMD, bar + DMA_CTRL_REG);
```

> 来源：`arch/x86/include/asm/io.h:88`
```c
#define writel_relaxed(v, a) __writel(v, a)
```
`writel` vs `writel_relaxed`：前者有屏障，后者无屏障。`iowrite32` 用的是 `writel`（有隐式写屏障），但posted写之间仍然可能乱序，所以需要显式 `wmb()`。

**wmb() 实际干了什么**：发 `mfence` 指令，flush store buffer，保证所有写都到达 PCIe 控制器后再继续执行。

## C. 常见错误案例

### 错误 1：ioremap 成功，但读写全 F

**现象**：ioremap() 返回非 NULL，但 readl() 返回 0xFFFFFFFF。

**排查**：检查 Command Register 的 Memory Space Enable 位（bit 1）是否为 1。

### 错误 2：64-bit BAR 申请失败

**现象**：BAR0 是 64-bit，但 BAR1 被另一个功能占用。

**原因**：64-bit BAR 必须连续（BAR0+BAR1），不能跳过 BAR1。

### 错误 3：DMA 描述符顺序错误（无 wmb）

**现象**：EP 先收到启动命令，后收到数据地址，DMA 传输错误数据。

**解决**：在描述符写入后、启动命令前加 wmb()。

## D. 决策树

```
配置 EP BAR 空间
├── BAR & 0x01 == 1
│   └── I/O BAR → 用 inb/outb（x86 专用）
├── BAR & 0x01 == 0 且 (BAR & 0x06) == 0x04
│   └── 64-bit MEM BAR → BAR0 存低位，BAR1 存高32位
└── BAR & 0x01 == 0 且 (BAR & 0x06) == 0x00
    └── 32-bit MEM BAR → 直接解析

访问 BAR 空间
├── pci_request_regions() → 向系统注册所有权
├── pci_iomap() 或 ioremap() → 建立虚拟地址映射
└── ioread32/iowrite32 → MMIO 读写，禁止直接解引用

写寄存器后加屏障
├── 写完一个 EP 寄存器，再写同一个 EP 另一个寄存器 → wmb()
├── 读完寄存器要依赖返回值 → rmb()
└── 既有读写又有依赖 → mb()
```
