## A. 原理（是什么）

### 为什么需要数据链路层？

PCIe 协议栈分为三层：

```
事务层（Transaction Layer）
    ↑
    │ TLP（Memory Read/Write/Cfg/Message）
    ↓
数据链路层（Data Link Layer）   ← 本节重点
    ↑
    │ DLLP（Ack/Nak/FC/Power）
    ↓
物理层（Physical Layer）
    ↑
    │ 串行差分信号（Gen1/2/3/4/5）
```

**事务层负责 TLP 的组装和解析。数据链路层负责：**
1. 对 TLP 添加 CRC（LCRC）和序列号
2. 发送 ACK/NACK，检测 TLP 传输错误并重传
3. 流量控制（Flow Control）：确保发送方不会淹没接收方

**没有数据链路层**：TLP 传输错误无法发现，丢包只能靠上层重传，效率极低。

### Flow Control 机制

PCIe 是点对点网络，没有共享总线。这意味着**每个接收方有有限的接收缓冲区**。

如果发送方无限制发送，接收方缓冲区会溢出，数据丢失。

**Flow Control 信用机制**解决此问题：

```
发送方（TX）                         接收方（RX）
   │                                    │
   │──── Request TLP ─────────────────→│（消耗 1 credit）
   │                                    │
   │←──── Flow Control DLLP (更新信用) ──│（接收方空闲了，信用恢复）
   │                                    │
```

**核心概念**：
- **Credit（信用）**：接收方告诉发送方"我还有多少缓冲空间"
- **Credit Limit**：接收方能接受的最大数据量
- **Credit Consumed**：发送方已经用了多少
- **更新机制**：接收方定期（或每次消费后）通过 DLLP 告诉发送方当前信用状态

**两种 Flow Control 类型**：

| FC 类型 | 管理的 TLP |
|---------|-----------|
| **Posted Request FC（P）** | MWr、Msg（已发送，不需要 Completion）|
| **Non-Posted Request FC（NP）** | MRd、CfgRd（需要 CplD/Cpl 返回）|
| **Completion FC（C）** | CplD、Cpl（用于返回数据或确认）|

**注意**：每种 FC 类型有独立的信用计数器，发送方必须同时维护三种信用。

---

### DLLP 类型

DLLP（Data Link Layer Packet）在相邻设备之间传递，不经过路由：

| DLLP 类型 | 作用 | 方向 |
|----------|------|------|
| **ACK** | 确认收到 TLP（序列号正确，CRC 正确）| Downstream → Upstream |
| **NAK** | 否定确认（CRC 错误或序列号错误），要求重传 | Downstream → Upstream |
| **FC_P** | 更新 Posted Request 信用 | 双向 |
| **FC_NP** | 更新 Non-Posted Request 信用 | 双向 |
| **FC_Cpl** | 更新 Completion 信用 | 双向 |
| **PM_Enter_L1** | 进入 L1 电源状态 | Downstream → Upstream |
| **PM_Enter_L0s** | 进入 L0s 电源状态 | Downstream → Upstream |
| **PM_ActiveState** | 确认电源状态已切换 | 双向 |

### DOE（Data Object Exchange）

DOE 是 PCIe 4.0 引入的带内配置协议，用于访问配置对象（Data Object）。

**典型用途**：
- **CMA（Configuration Manager Access）**：读取和写入 PCIe 扩展配置空间
- **IDE（In-band Debug）**：用于调试和错误注入
- **VSEC（Vendor-Specific Extended Capability）**：厂商自定义配置

**DOE 交易类型**：
- **DOE Read**（类似 MRd → CplD）：发起读请求，接收方返回数据
- **DOE Write**（类似 MWr）：直接写入数据对象

**DOE 在内核中的实现**：`drivers/pci/doe.c`，核心结构：

源码（`drivers/pci/doe.c:38`）：
```c
struct pci_doe_mb {
    struct pci_dev *pdev;        // 所属 PCI 设备
    u16             cap_offset;  // DOE Capability 在配置空间的偏移
    struct xarray   feats;       // 支持的 DOE Feature 数组
    wait_queue_head_t wq;        // 等待队列
    struct workqueue_struct *work_queue;  // 工作队列
    unsigned long   flags;       // PCI_DOE_FLAG_* 状态标志
};

struct pci_doe_feature {
    u16 vid;     // Vendor ID
    u8  type;    // Data Object Type
};

struct pci_doe_task {
    struct pci_doe_feature feat;  // 要访问的 Feature
    void   *request_pl;          // 请求载荷
    size_t  request_pl_sz;        // 请求载荷大小
    void   *response_pl;          // 响应载荷
    size_t  response_pl_sz;        // 响应载荷大小
    size_t  rv;                   // 返回值（接收到的响应长度）
    void (*complete)(struct pci_doe_task *task);  // 完成回调
    struct pci_doe_mb *doe_mb;    // 所属 Mailbox
    // ...
};

// 关键 API（drivers/pci/doe.c）
pci_doe_create_mb(dev, cap_offset);  // 创建 DOE Mailbox
pci_doe_write(mb, &task);            // 发起 DOE Write
pci_doe_read(mb, &task);              // 发起 DOE Read
```

## B. 为什么这样设计

### 为什么 DLLP 是邻居之间传递，而不是路由？

数据链路层是相邻-hop 协议。只有直接相连的两个端口交换 DLLP，不需要知道整条路径上有哪些设备。

类比：两台相邻路由器之间交换流量控制帧，不需要端到端信令。

### 为什么 ACK/NAK 要基于序列号？

TLP 在链路上可能乱序（特别是经过 Switch），接收方用序列号识别"这是第几个 TLP"，从而检测丢失或重复。

### 为什么 FC 更新是定期的而不是每次都更新？

如果每次 TLP 传输都触发 FC DLLP，开销太大。通常按时间周期或信用阈值更新，平衡效率和可靠性。

## C. 常见错误案例

### 错误 1：接收方缓冲区溢出（FC 信用耗尽）

**现象**：链路训练成功，但 EP 设备无法接收 MWr，持续报 FC 信用不足。

**排查**：`lspci -vv` 查看 AER 状态，确认链路速率和宽度正确。检查 FC 协商参数。

### 错误 2：Switch 上游收到 NAK 导致性能下降

**现象**：大量重传导致带宽利用率下降。

**排查**：检查物理链路质量（线缆、连接器）；检查是否有信号完整性问题。

### 错误 3：DOE 通信超时

**现象**：`pci_doe_write/read` 返回 -ETIMEDOUT`。

**排查**：确认对端 DOE 对象存在；检查 DOE 协议规范中对象的数据格式是否正确。

## D. 决策树

```
发送 TLP 后收到 NAK
  → 重传该 TLP（序列号对应的那一帧）
  → 检查物理链路质量

FC 信用耗尽（发送方无法发送）
  → 等待 FC Update DLLP
  → 检查接收方是否正常工作

链路建立后 DLLP 无法交换
  → 检查 Flow Control 协商参数
  → 可能是协商失败导致 L0s 状态异常

DOE 读写出错
  → 确认对象 VID/DOE Index 正确
  → 确认数据格式符合 DOE 协议规范
```
