# S1_03_同步原语

## 一、spinlock（自旋锁）

```c
spinlock_t lock;
spin_lock(&lock);
// 临界区
spin_unlock(&lock);
```

**为什么叫自旋**：获取不到锁时 CPU 忙等待，不切换任务。

**为什么不能睡眠**：spinlock 禁用了内核抢占，持锁时如果调用可能睡眠的函数，会死锁。

**`spin_lock_irqsave`**：
```c
unsigned long flags;
spin_lock_irqsave(&lock, flags);   // 1.关中断 2.关抢占 3.加锁
// 临界区
spin_unlock_irqrestore(&lock, flags); // 1.解锁 2.开抢占 3.开中断
```

为什么要关中断？因为中断处理函数也可能访问同一把锁。

## 二、mutex（互斥锁）

| 维度 | spinlock | mutex |
|------|---------|-------|
| 等待方式 | 忙等待（占CPU）| 睡眠等待（不占CPU）|
| 适用场景 | 临界区短 | 临界区长 |
| 中断上下文 | ✅ 可以 | ❌ 不可以 |

```c
mutex_lock(&drv_data->mutex);
// 这里可以睡眠，如 kmalloc(GFP_KERNEL)
mutex_unlock(&drv_data->mutex);
```

## 三、completion（一次性信号量）

用于等一个事件完成（一次性）：

```c
static DECLARE_COMPLETION(done);

// 等待方
wait_for_completion(&done);

// 触发方（在中断中）
complete(&done);
```

**典型场景**：模块卸载时等待所有操作完成。

## 四、死锁案例

**Case 1：中断和进程竞争同一把锁**

```c
// 进程上下文，没关中断
spin_lock(&lock);
dev->reg = val;
spin_unlock(&lock);  // 若此时来中断 → 中断也等这把锁 → 死锁

// 修复：用 spin_lock_irqsave
spin_lock_irqsave(&lock, flags);
dev->reg = val;
spin_unlock_irqrestore(&lock, flags);
```

**Case 2：ABBA deadlock（加锁顺序不一致）**

```c
// func1: 先A后B
spin_lock(&lock_a);
spin_lock(&lock_b);

// func2: 先B后A（不同顺序！）
spin_lock(&lock_b);  // 如果先拿到B
spin_lock(&lock_a);  // 等A，但A被func1持有着 → 死锁
```

**修复**：所有代码路径用相同的加锁顺序。

## 五、lockdep 死锁检测

编译内核时启用 `CONFIG_PROVE_LOCKING=y`，lockdep 自动检测：
- 递归加锁
- ABBA 死锁
- 错误锁类型使用

## 六、选择决策树

```
是中断处理函数吗？
  → 是 → spinlock_irqsave

持锁时间长吗（需要睡眠）？
  → 是 → mutex
  → 否 → spinlock

等一次性事件？
  → 是 → completion
```

## 七、PCIe 驱动典型用法

```c
static spinlock_t regs_lock;

// 中断里：必须 spinlock_irqsave
static irqreturn_t irq_handler(int irq, void *dev)
{
    spin_lock_irqsave(&regs_lock, flags);
    shared_value = readl(bar + 0x10);
    spin_unlock_irqrestore(&regs_lock, flags);
}

// 用户态 read：同样用 spinlock_irqsave
static ssize_t read(...) {
    spin_lock_irqsave(&regs_lock, flags);
    val = shared_value;
    spin_unlock_irqrestore(&regs_lock, flags);
}
```
