# S1_02_内核内存管理

## 一、三种内存申请的选择

| 函数 | 物理连续 | 适用场景 |
|------|---------|---------|
| `kmalloc` | ✅ 是 | 驱动数据结构（<128KB） |
| `vmalloc` | ❌ 否 | 大缓冲、临时缓冲（>128KB） |
| `ioremap` | N/A | **映射设备寄存器**，不是分配内存 |

**记忆法**：
- kmalloc = kernel allocate = 普通内存分配
- vmalloc = virtual = 虚拟大空间，物理不连续
- ioremap = I/O remap = 把设备寄存器地址映射到CPU可访问的虚拟地址

## 二、GFP_* 标志

```c
void *kmalloc(size_t size, gfp_t flags);
```

| 标志 | 含义 | 使用场景 |
|------|------|---------|
| `GFP_KERNEL` | 允许睡眠 | 进程上下文，驱动初始化 |
| `GFP_ATOMIC` | 不允许睡眠 | 中断处理函数、spinlock保护的代码 |
| `GFP_NOIO` | 允许回收但不触发IO | 块设备底层 |
| `GFP_NOFS` | 允许回收但不触发FS | 文件系统代码 |

**为什么中断里不能用 GFP_KERNEL**：中断上下文不能睡眠。GFP_KERNEL 内部可能启动IO回收内存，IO会触发睡眠等待，导致死锁。

## 三、ioremap 的正确使用

**ioremap 不是分配内存，是映射设备寄存器！**

```c
void __iomem *bar = ioremap(pci_addr, size);

// 错误：直接解引用
*bar = 0x1234;              // ❌ 可能导致总线错误
*(volatile u32*)bar = 0x1234; // ❌ 同样危险

// 正确：用读写函数
writel(0x1234, bar);         // ✅
u32 val = readl(bar + 0x04); // ✅
```

**ioremap 必须和 iounmap 配对**（或用 `devm_ioremap` 自动管理）。

## 四、devm_* 函数（推荐）

`devm_` = managed，自动在 remove 时释放资源。

```c
// 传统：remove 中必须手动 iounmap
void remove(...) { iounmap(bar); }

// devm：内核自动处理，不需要 remove 中调用
void probe(...) { bar = devm_ioremap(&dev->dev, addr, size); }
```

**原则**：能用 `devm_` 就用 `devm_`，减少错误。

## 五、内存屏障

```c
writel(0x1, ctrl_reg);  // 写控制寄存器
wmb();                   // 确保写操作完成后再往下
writel(0x1, trigger);  // 写触发寄存器
```

**为什么需要**：CPU 会重排序读写操作。外设寄存器有副作用，必须按顺序执行。

| 函数 | 作用 |
|------|------|
| `mb()` | 全部读写屏障 |
| `rmb()` | 读屏障 |
| `wmb()` | 写屏障 |

## 六、决策树

```
遇到内存申请：

1. 是设备寄存器（MMIO）吗？
   → 是 → devm_ioremap / ioremap
   → 否 → 继续

2. 需要物理连续吗？
   → 是 → kmalloc / devm_kmalloc
   → 否 → vmalloc

3. 可以睡眠吗？
   → 进程上下文 → GFP_KERNEL
   → 中断上下文 → GFP_ATOMIC
```

## 七、常见错误

- ❌ 中断里用 GFP_KERNEL → "BUG: scheduling while atomic"
- ❌ ioremap 后直接解引用 → 总线错误
- ❌ 忘记 iounmap → 地址空间泄漏（用 devm 避免）
- ❌ 忘记 kfree → 内存泄漏（用 devm_kmalloc 避免）
