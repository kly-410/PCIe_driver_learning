## A. 原理（是什么）

### Linux PCI 子系统初始化三阶段

```
1. BIOS/UEFI 阶段
   → 建立 MCFG 表（ACPI）
   → RC 固件配置 ECAM 基址

2. 内核 PCI 子系统初始化（pci_scan_root_bus）
   → 扫描 RC 下游所有 Bus
   → 发现 EP 设备，分配 Bus/Dev/Func
   → 分配 MMIO 地址（BAR）

3. 驱动注册和 probe
   → pci_driver.id_table 和 pci_dev 对应关系
   → 匹配成功 → probe() 被调用
   → 匹配失败 → 设备无驱动
```

### `struct pci_host_bridge` — RC 的内核抽象

```c
struct pci_host_bridge {
    struct device dev;           // 设备模型基类
    struct list_head windows;    // 地址窗口（BAR 分配后的 MMIO 范围）
    resource_size_t mmio_alt_base; // 备选 MMIO 基址
    int domain;                   // PCI 域编号
    int busnr;                   // 起始总线号
    void (*release_fn)(...);     // 释放函数
    // ...
};
```

`pci_host_bridge` 是 RC 的软件抽象。`pci_scan_root_bus()` 创建这个 bridge 并扫描下游 Bus。

### `struct pci_dev` — 每个 PCIe 设备的内核抽象

```c
struct pci_dev {
    struct pci_bus *bus;         // 所属总线
    unsigned int devfn;         // [Bus<<3 | Dev<<1 | Func] 编码
    u16 vendor;
    u16 device;
    u8  hdr_type;              // 0x00=EP, 0x01=Bridge
    struct pci_driver *driver;   // 绑定的驱动
    struct resource resource[6]; // BAR 对应的地址范围（Linux resource 管理）
    // ...
};
```

### BAR 分配流程

```
1. pci_scan_root_bus() 扫描总线，发现 EP
2. 为 EP 分配 Bus/Dev/Func
3. pci_enable_device() 使能设备
4. pci_bar_init() 读取 BAR 值
5. pci_assign_resource() 分配 MMIO 地址（从系统可用地址空间）
6. pci_write_config_word(BAR, addr) 写入 BAR
```

### 内核模块基本结构

```c
#include <linux/pci.h>
#include <linux/module.h>

/* 驱动名称（和 id_table 对应）*/
#define DRIVER_NAME "my_pci_driver"

/* 支持的设备列表（PCI ID 表）*/
static const struct pci_device_id ids[] = {
    { PCI_DEVICE(0x8086, 0x1234) },  /* Intel, Device 0x1234 */
    { 0, }  /* 结束标记 */
};
MODULE_DEVICE_TABLE(pci, ids);

/* probe: 驱动匹配到设备后调用 */
static int probe(struct pci_dev *dev, const struct pci_device_id *id)
{
    printk(KERN_INFO "my_pci: probe device %s\n", pci_name(dev));

    /* 使能设备（使能 PCI MMIO 和 IO）*/
    pci_enable_device(dev);

    /* 申请 BAR 地址区域 */
    if (pci_request_regions(dev, DRIVER_NAME))
        return -EBUSY;

    /* 读取 BAR0（MMIO 基址）*/
    void __iomem *bar0 = pci_iomap(dev, 0, 0);
    if (!bar0) {
        pci_release_regions(dev);
        return -ENOMEM;
    }

    return 0;
}

/* remove: 驱动卸载或设备移除时调用 */
static void remove(struct pci_dev *dev)
{
    pci_iounmap(dev, bar0);
    pci_release_regions(dev);
}

/* pci_driver 描述符 */
static struct pci_driver my_pci_driver = {
    .name     = DRIVER_NAME,
    .id_table = ids,
    .probe    = probe,
    .remove   = remove,
};

/* 模块入口 */
static int __init my_init(void)
{
    return pci_register_driver(&my_pci_driver);
}

/* 模块出口 */
static void __exit my_exit(void)
{
    pci_unregister_driver(&my_pci_driver);
}

module_init(my_init);
module_exit(my_exit);
MODULE_LICENSE("GPL");
```

## B. 为什么这样设计

### 为什么驱动和设备要分开注册？

PCIe 设备是硬件，驱动是软件，内核不知道系统里有哪些设备（取决于主板和插了什么卡）。所以：
1. **PCI 子系统先扫描设备**（pci_scan_bus），把所有 pci_dev 加入设备链表
2. **驱动注册自己**（pci_register_driver），带上 id_table
3. **匹配成功**则调用 probe

这种"设备先行、驱动后连"的机制允许不重启系统就动态加载驱动（modprobe）。

### 为什么 BAR 分配在内核 probe 阶段？

BIOS 阶段可能没有给所有 BAR 分配地址（特别是 Linux 自己接管 PCI 总线时）。Linux 重新扫描时需要重新分配，这是 PCI 子系统重新初始化的核心。

### `pci_enable_device` 做了什么？

- 将 Command Register 的 I/O Space Enable 和 Memory Space Enable 位置位
- 设备开始响应 MMIO 和 IO 访问
- 如果设备还没分配 BAR，enable 后仍然无法访问 MMIO

## C. 常见错误案例

### 错误 1：probe 里没调用 pci_enable_device，BAR 访问返回全 F

**现象**：读取 BAR 映射地址，总是返回 0xFFFFFFFF。

**排查**：确认 probe 入口调用了 `pci_enable_device(dev)`。

### 错误 2：pci_iounmap 用了错误的 bar 地址

**现象**：remove 时 kernel panic。

**排查**：pci_iounmap 的地址必须和 pci_iomap 返回值完全一致。

### 错误 3：id_table 没有加 MODULE_DEVICE_TABLE，modprobe 找不到驱动

**现象**：驱动加载了，但设备不自动匹配 probe。

**排查**：确认有 `MODULE_DEVICE_TABLE(pci, ids)` 宏。

## D. 决策树

```
驱动加载 → pci_register_driver()
  → id_table 和 pci_dev 匹配
      → probe() 被调用
          → pci_enable_device(dev) ✓
          → pci_request_regions() ✓
          → pci_iomap(dev, bar) ✓
          → 正常访问设备

      → 不匹配 → 设备留在"无驱动"状态

设备移除 → pci_remove_bus_device()
  → 驱动 remove() 被调用
      → pci_iounmap()
      → pci_release_regions()
      → 资源释放
```
