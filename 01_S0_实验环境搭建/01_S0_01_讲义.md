# S0_01 — 实验环境搭建

## 本节目标

- 在 Ubuntu 24.04 上从源码编译 QEMU 9.0（aarch64 版本，支持 PCIe 完整模拟）
- 配置 Linux 内核源码（6.17），能够编译 PCI 驱动模块
- 验证 QEMU virt 平台的 PCIe RC 模拟正常（ECAM + MSI + AER）
- 建立开发目录结构，熟悉实验流程

## 前置依赖

无。本节是所有实验的起点。

---

## A. 原理（是什么）

### QEMU virt 作为 PCIe 开发平台

QEMU virt 机器模型提供完整的 PCIe RC 模拟：

| 模拟组件 | 说明 |
|---------|------|
| **ECAM** | 内存映射配置空间（默认基址 0xef800000，256MB 窗口）|
| **MSI** | Message Signaled Interrupt，支持多向量 |
| **AER** | Advanced Error Reporting，错误注入和报告 |
| **ATU** | Address Translation Unit，Outbound/Inbound 地址窗口 |

### 为什么选 QEMU virt（ARM 平台）？

| 方案 | 优点 | 缺点 |
|------|------|------|
| **QEMU virt (aarch64)** | PCIe RC 完整模拟，设备模型干净，可改寄存器 | TCG 软件模拟速率（无硬件加速） |
| **QEMU q35/pc (x86_64)** | x86 硬件虚拟化加速（KVM） | virt 机器类型不存在，q35 模型复杂 |
| **真实 x86 PC** | 真实 Gen4 速率 | 不能改 Intel RC 驱动 |
| **树莓派 4** | 真实硬件，可写 RC 驱动 | 只有一个端口，调试手段少 |

**virt 是 ARM 的机器类型**，不是 x86_64。这意味着：
- 必须编译 `aarch64-softmmu` 版本
- 不能使用 x86_64 的 qemu-system-x86_64
- 在桌面 CPU 上使用 TCG 软件模拟（无 KVM 加速）

## B. 为什么这样设计

### 为什么要从源码编译 QEMU 9.0？

Ubuntu 24.04 官方 apt 源只有 QEMU 8.2。QEMU 8.2 对 PCIe ARIF（Alternative Routing-ID Forwarding）实现有 bug，导致某些多 endpoint 场景下路由行为异常。QEMU 9.0 修复了这个问题。

### 为什么要编译 aarch64 版本？

`virt` 是 ARM 的机器类型，x86_64 版本的 qemu-system-x86_64 不支持 `-M virt`。必须单独编译 aarch64-softmmu 版本。

### 为什么用 Ubuntu 24.04？

GCC 13 / GDB 14 / QEMU 9.0 可直接 apt 安装，HWE 内核 6.17 与 headers 完全匹配，避免版本错配。

### 为什么需要 linux-headers 而不是 linux-source？

linux-headers 只包含编译驱动所需的 Makefile、头文件（include/）、Kconfig，不包含完整源码树（几百MB）。驱动编译只需要知道"内核导出的 API 长什么样"，不需要内核自身的实现代码。

## C. 常见错误案例

### 错误 1：QEMU 编译缺少依赖

```
ERROR: glib-2.48 gthread-2.0 not found
```
**解决**：`sudo apt install build-essential libglib2.0-dev libpixman-1-dev libncurses-dev libssl-dev python3 ninja-build`

### 错误 2：内核模块编译失败（Makefile 用空格而非 Tab）

```
Makefile:*** missing separator
```
**原因**：Makefile 的 recipe 行必须以 Tab 开头，不能用空格。
**解决**：确认 recipe 行以 Tab 字符开头。

### 错误 3：QEMU 下载慢（官方源）

官方 download.qemu.org 在国内下载极慢（约 1 小时+）。
**解决**：换国内镜像：
```bash
#### 阿里云（不一定有所有版本）
sudo wget https://mirrors.aliyun.com/qemu/qemu-9.0.0.tar.xz
#### 腾讯云
sudo wget https://mirrors.cloud.tencent.com/qemu/qemu-9.0.0.tar.xz
#### 华为云
sudo wget https://mirrors.huaweicloud.com/qemu/qemu-9.0.0.tar.xz
```
如果镜像没有 9.0.0 版本，只能用官方源加 `-c` 断点续传，或者先用 apt 版 8.2 临时学习。

### 错误 4：x86_64 版本用 `-M virt` 报错

```
qemu-system-x86_64: unsupported machine type
Use -machine help to list supported machines
```
**原因**：`virt` 是 ARM (aarch64) 专用的机器类型，x86_64 版本没有这个机器。
**解决**：单独编译 aarch64-softmmu 版本：
```bash
cd /opt/qemu-9.0.0
sudo ./configure --target-list=aarch64-softmmu
sudo make -j$(nproc)
sudo make install
qemu-system-aarch64 --version
```

### 错误 5：KVM 不支持 aarch64

```
qemu-system-aarch64: invalid accelerator kvm
```
**原因**：桌面 CPU 通常没有 ARM 硬件虚拟化扩展（ARM Virtuaization Extensions），所以 aarch64 的 KVM 加速不可用。
**解决**：去掉 `-enable-kvm`，使用 TCG 软件模拟：
```bash
qemu-system-aarch64 -M virt -m 256M -display none -nographic
#### 不加 -enable-kvm
```

### 错误 6：configure / make 在 build 子目录执行

如果 configure 之后 build/ 子目录里找不到 aarch64 相关产物，检查是否在 build/ 子目录里执行了 make。
**解决**：configure 和 make 都在源码根目录执行，不要 cd 到 build/：
```bash
cd /opt/qemu-9.0.0        # 源码根目录，不是 build/
sudo ./configure ...
sudo make -j$(nproc)
sudo make install
```

### 错误 7：QEMU 安装路径不在 PATH 里

从源码编译安装的 QEMU 在 `/usr/local/bin/`，不在 `/opt/qemu/bin/`。
**验证**：`which qemu-system-aarch64` 或 `/usr/local/bin/qemu-system-aarch64 --version`

## D. 决策树

```
QEMU 环境问题
├── 官方源下载太慢
│   └── 换国内镜像（阿里云/腾讯云/华为云）
│       └── 镜像没有该版本 → 用官方源 + -c 断点续传
│
├── 报错 unsupported machine type 'virt'
│   └── 用 qemu-system-x86_64 → 换成 qemu-system-aarch64
│       └── aarch64 版本未安装 → 源码编译 aarch64-softmmu
│
├── -enable-kvm 报错 invalid accelerator
│   └── aarch64 平台不支持 KVM（桌面 CPU 无 ARM 虚拟化）
│       └── 去掉 -enable-kvm，使用 TCG 软件模拟
│
├── configure 后 make 产物不对
│   └── 检查是否在 build/ 子目录执行了 make
│       └── 回到源码根目录重新 configure + make
│
├── 找不到 qemu-system-aarch64 命令
│   └── 检查 /usr/local/bin/（源码安装路径）
│       └── 未安装 → make install
│
├── lspci 看不到设备
│   └── 检查 QEMU 设备是否正确挂载（-device 参数）
│
├── MSI 中断不触发
│   └── 检查 QEMU 版本（需 9.0+），检查 MSI Cap
│
├── 编译内核模块失败
│   ├── Makefile separator 问题 → 确认 Tab 字符
│   └── headers 版本不匹配 → 确认 uname -r 对应 headers
```

## E. 补充：代码编译到 home 目录（NTFS 分区执行权限问题）

**问题**：课程代码放在 `/mnt/software/` 等 NTFS 分区时，`chmod +x` 不生效（NTFS 不支持 Linux 执行权限），导致编译出来的可执行文件无法运行。

**解决**：把课程代码整体迁移到 home 目录，后续所有实验在 home 目录下编译和运行：

```bash
#### 把仓库从 NTFS 分区移到 home 目录
mv /mnt/software/code/PCIe_driver_learning ~/work/code/

#### 以后进 home 目录的仓库
cd ~/work/code/PCIe_driver_learning/02_S1_PCIe总线基础与TLP/
make scan && sudo ./scan
```

**Makefile 规范**：每个编译规则后必须加 `chmod +x`，确保编译产物有执行权限：
```makefile
scan: S1_01_pci_scan.c
    $(CC) $(CFLAGS) -o $@ $< $(LDFLAGS)
    chmod +x $@
```
