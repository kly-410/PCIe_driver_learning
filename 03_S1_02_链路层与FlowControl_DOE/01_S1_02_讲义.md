# S1_02 — 链路层与 Flow Control + DOE

## 本节目标

## A. 原理（是什么）

### 链路层的位置

```
┌─────────────────┐
│ Transaction Layer│  TLP (MRd/MWr/CfgRd)
└────────┬────────┘
         ▼
┌─────────────────┐
│   Data Link     │  DLLP (Ack/Nak/FC)
│     Layer       │  + LCRC + Seq#
└────────┬────────┘
         ▼
┌─────────────────┐
│  Physical Layer │  8b/10b or 128b/130b encoding
└─────────────────┘
```

链路层在 TLP 外面包了一层：
- **发送端**：加 LCRC（32-bit）+ Sequence Number（4-bit，0~15）
- **接收端**：校验 LCRC → 正确回 Ack，错误回 Nak
- **错误处理**：发送方从 Nak 对应序号开始重发整个窗口（最大 16 TLP）

### DLLP 类型

| DLLP 类型 | 用途 |
|----------|------|
| **Ack** | 确认收到 TLP Seq#N |
| **Nak** | 要求重传（从 Seq#N-1 开始）|
| **PM\_Active/PM\_L0s/PM\_L1** | 电源管理状态切换 |
| **InitFC1 / InitFC2** | 初始化 Flow Control Credit |
| **UpdateFC** | 运行时刷新 Credit |
| **VendorDefined** | 厂商自定义（包括 DOE）|

### Flow Control Credit

接收方 buffer 有限，FC 机制让接收方定期告知发送方"我还剩多少 Credit"。

| Credit 类型 | 对应资源 | 最小粒度 |
|------------|----------|---------|
| **PH** | Posted 请求头 | 1 TLP |
| **PD** | Posted 数据负载 | 4 DW |
| **NPH** | Non-Posted 请求头 | 1 TLP |
| **NPD** | Non-Posted 数据负载 | 4 DW |
| **CPLH** | 响应头 | 1 TLP |
| **CPLD** | 响应数据 | 4 DW |

### DOE（Data Object Exchange）

DOE 是 PCIe Spec 6.x 引入的带内配置协议，用于访问 VPD、CIS、CMM 等大数据块。

- **ECAM**：直接地址访问配置空间（4KB/func），适合寄存器
- **DOE**：Mailbox 协议，适合大数据 + 握手机制

DOE Mailbox 结构：写 DOE_OBJECT_HEADER（vendor\_id + object\_type）→ 轮询 DOE\_INT\_STATUS 的 DONE 位 → 读响应数据。

## B. 为什么这样设计

### 为什么需要链路层？

TLP 在链路上可能位翻转（噪声）、丢失（buffer 溢出）、乱序。链路层通过 ACK/NACK 机制保证端到端的可靠传输，发送方保留已发 TLP 的副本（Replay Buffer），直到收到 Ack 才删除。

### 为什么区分 Posted 和 Non-Posted？

Posted 写不阻塞 RC，可流水线发送提高性能。Non-Posted 读必须等数据返回，本质是被动的——RC 不知道 EP 存了什么，必须等 EP 主动返回。

### FC 为什么定期刷新？

接收方 buffer 动态变化（消费 Credit），如果 FC 不刷新，发送方不知道接收方还有多少 Credit。PCIe 规定 UpdateFC DLLP 至少每 30us 发送一次，防止发送方误以为 Credit 充足而超发。

## C. 常见错误案例

### 错误 1：Flow Control Credit 超额

**现象**：链路偶尔丢 TLP，`dmesg` 报 `device link may not be reliable` 警告。

**排查**：
```bash
# 查看链路速率和协商宽度
lspci -vv -nn -s 00:00.0 | grep LnkSta
# 对比 RC 和 EP 的 Max_Payload_Size
setpci -s 00:00.0 DEVICE_CAPS  # EP 的 Max_Payload_Size
```

**原因**：Max\_Payload\_Size 和接收方 buffer 不匹配。强制降到 Gen1/x1 可以验证。

### 错误 2：DOE Mailbox 超时

**现象**：`pci_doe_write()` 返回 `-ETIMEDOUT`。

**排查**：
```bash
# 确认 DOE Cap 存在
setpci -s XX:XX.X DOE_CAP_HDR
# 读 Interrupt Status
setpci -s XX:XX.X DOE_INT_STATUS
```

## D. 决策树

```
链路训练完成后链路不稳定
├── 丢 TLP，报 "device link may not be reliable"
│   └── FC Credit 问题 → 检查两端 Max_Payload_Size 是否匹配
├── 链路反复进入 Recovery
│   └── 物理层问题 → 检查 RefClk / SerDes 配置 / Lane 极性
└── MSI 中断不触发
    └── DOE Mailbox 问题 → 检查 DOE Cap + Interrupt Status
```
