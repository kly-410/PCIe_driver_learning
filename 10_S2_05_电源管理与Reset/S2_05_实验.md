# S2_05_实验 — 电源管理 + Fundamental Reset + NPEM

---

## 实验 1：查看设备电源状态

```bash
### 查看 01:00.0 的电源状态
cat /sys/bus/pci/devices/0000:01:00.0/power/runtime_status
```

## 实验 2：手动触发 FLR（如果硬件支持）

```bash
### FLR（Function Level Reset）
echo 1 | sudo tee /sys/bus/pci/devices/0000:01:00.0/reset
```

### 改错题（Challenge）

代码中故意埋了一个 bug，运行或审查后找出问题。

**实验代码**（`S2_05_power.c`，在 `10_S2_05_电源管理与Reset/` 目录下）：

```c
/* S2_05_power.c —— 故意埋了 D3→D0 转换顺序错误 bug */
#include <linux/pci.h>
#include <linux/module.h>

static int my_probe(struct pci_dev *dev, const struct pci_device_id *id)
{
    int ret;

    dev_info(&dev->dev, "Probing device, current power state unknown\n");

    /* 尝试把设备从 D3hot 转换到 D0（工作状态）*/
    /* 错误的顺序：先写 0x00 再写 0x03，实际序列应该是 ... */

    /* 读取当前 PMCSR（Power Management Control/Status）*/
    u16 pmcsr = pci_read_config_word(dev, dev->pm_cap + PCI_PM_CTRL);
    dev_info(&dev->dev, "PMCSR = 0x%04x\n", pmcsr);

    /* 错误的 D3→D0 转换序列： */
    /* D3hot → D0 的正确顺序是：D3hot → D0，不需要经过 D3cold？ */
    /* 实际上 PCIe 规范：PM 状态转换顺序有约束，见下表 */

    /* 错误：直接从 D3hot 跳到 D0 without proper sequence */
    /* PCIe Spec 说：D3hot → D0 是直接转换，但有些设备需要先进入 D3cold 再 D0 */
    /* 下面的代码跳过了必要的步骤：*/

    /* 步骤1：把 Power State 字段从 D3hot(11b) 写成 D0(00b) */
    pci_write_config_word(dev, dev->pm_cap + PCI_PM_CTRL, 0x0000);
    /* ← 直接写 D0，但没等待安规时间（100ms）+ 延迟应答 */

    /* 步骤2：立即访问 BAR（此时设备可能还没完成转换！）*/
    msleep(10);  /* 只等了 10ms，但 Spec 要求至少 100ms */
    u32 val = ioread32(pci_iomap(dev, 0, 0) + 0x00);  /* 可能读到全 F 或不确定值 */

    dev_info(&dev->dev, "Read BAR0 after D3→D0: 0x%08x\n", val);

    return 0;
}

static void my_remove(struct pci_dev *dev)
{
    /* 从 D0 → D3hot：直接写 PMCSR */
    u16 pmcsr = pci_read_config_word(dev, dev->pm_cap + PCI_PM_CTRL);
    pmcsr &= ~PCI_PM_STATE_MASK;
    pmcsr |= PCI_D3hot;
    pci_write_config_word(dev, dev->pm_cap + PCI_PM_CTRL, pmcsr);
}

static struct pci_driver my_driver = {
    .name = "s2_05_power",
    .probe = my_probe,
    .remove = my_remove,
};

module_pci_driver(my_driver);
MODULE_LICENSE("GPL");
```

**问题**：从 D3hot 转换到 D0（恢复工作）时，转换序列有严格要求：

| 转换 | 是否需要等待 |
|:---|:---|
| D0 → D3hot | 无需等待 |
| D3hot → D0 | 必须等待 100ms（PCIe Spec Section 6.6.1）|
| D3cold → D0 | 必须等待 1s（主电源重新使能）|

1. 找到代码中 `msleep(10)` 的问题（等待时间不足）
2. 写出正确的 D3hot → D0 转换序列

> 提示：PCIe Spec 要求 D3hot → D0 之间至少 100ms 等待（PMCSR 写完后）。部分老旧设备还需要额外延迟。此外，`pci_set_power_state(dev, PCI_D0)` 是内核提供的标准函数，应该用它而不是直接写 PMCSR。

预期答案：
1. `msleep(10)` 应改为 `msleep(100)` 或更长
2. 正确做法是调用内核 API：`pci_set_power_state(dev, PCI_D0)`，它会自动处理延迟和状态转换顺序，而不是手动写 PMCSR
3. 当前代码直接写 PMCSR 忽略了状态转换完成标志（可能需要 poll `PMCSR` 的 `PS` 字段确认转换完成）
