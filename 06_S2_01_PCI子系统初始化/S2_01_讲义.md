## A. 原理（是什么）

### Linux PCI 子系统初始化三阶段

```
1. BIOS/UEFI 阶段
   → 建立 MCFG 表（ACPI）
   → RC 固件配置 ECAM 基址

2. 内核 PCI 子系统初始化（pci_scan_root_bus）
   → 扫描 RC 下游所有 Bus
   → 发现 EP 设备，分配 Bus/Dev/Func
   → 分配 MMIO 地址（BAR）

3. 驱动注册和 probe
   → pci_driver.id_table 和 pci_dev 对应关系
   → 匹配成功 → probe() 被调用
   → 匹配失败 → 设备无驱动
```

### `do_pci_enable_device` — 设备使能内部实现

源码：`drivers/pci/pci.c:1981`

```c
static int do_pci_enable_device(struct pci_dev *dev, int bars)
{
    int err;
    u16 cmd;
    u8 pin;

    // 1. 把设备置入 D0 电源状态
    err = pci_set_power_state(dev, PCI_D0);
    if (err < 0 && err != -EIO)
        return err;

    // 2. 上游 Bridge 的 PCIe ASPM 功耗配置
    bridge = pci_upstream_bridge(dev);
    if (bridge)
        pcie_aspm_powersave_config_link(bridge);

    // 3. 调用平台特定的使能回调（pcibios_enable_device）
    err = pcibios_enable_device(dev, bars);
    if (err < 0)
        goto err_enable;

    // 4. 调用 PCI 修复钩子（让固件做最后配置）
    pci_fixup_device(pci_fixup_enable, dev);

    // 5. 如果不支持 MSI/MSI-X，读 INTx 引脚
    if (dev->msi_enabled || dev->msix_enabled)
        return 0;
    pci_read_config_byte(dev, PCI_INTERRUPT_PIN, &pin);
    // ...
}
```

### BAR 分配流程

```
pci_scan_root_bus() 扫描总线，发现 EP
  ↓
pci_read_config() 读 BAR 值（0xFFFFFFFF = 未分配）
  ↓
pci_assign_resource() 从系统可用地址空间分配一个窗口
  ↓
pci_write_config(BAR, addr) 写入 BAR 寄存器
  ↓
驱动 probe 里 pci_iomap(dev, bar) 建立 MMIO 映射
```

### 内核模块基本结构

```c
/*
 * S2_01_pci_skeleton.c - 最小可运行 PCI 驱动骨架
 * 源码：~/work/code/linux-source/
 */
`#include <linux/pci.h>     /* PCI 驱动框架（include/linux/pci.h）*/`
`#include <linux/module.h>  /* MODULE_* 宏 */`
`#include <linux/init.h>     /* __init / __exit 标记 */`

`#define DRIVER_NAME "S2_01_pci_skeleton"`

/* id_table：驱动支持哪些设备
 * PCI_DEVICE(vendor, device) 展开后：
 *   .vendor = vendor, .device = device
 *   .subvendor = PCI_ANY_ID, .subdevice = PCI_ANY_ID（匹配任何子系统）
 */
static const struct pci_device_id ids[] = {
    { PCI_DEVICE(PCI_ANY_ID, PCI_ANY_ID) },  /* 匹配所有设备 */
    { 0, }  /* 结束标记，必须有 */
};
MODULE_DEVICE_TABLE(pci, ids);  /* 让 modprobe 自动发现本驱动 */

/*
 * probe - 设备和驱动匹配成功后调用（drivers/pci/pci-driver.c:1464）
 *
 * 关键调用顺序：
 *   1. pci_enable_device(dev)   ← 必须先调用，使能 MMIO
 *   2. pci_request_regions()    ← 申请 BAR I/O 范围
 *   3. pci_iomap(dev, bar)      ← 建立 MMIO 虚拟地址映射
 */
static int probe(struct pci_dev *dev, const struct pci_device_id *id)
{
    printk(KERN_INFO "%s: probe() called for %s\n",
           DRIVER_NAME, pci_name(dev));  /* pci_name = "0000:01:00.0" */

    if (pci_enable_device(dev)) {         /* 使能 PCI MMIO + IO */
        dev_err(&dev->dev, "pci_enable_device failed\n");
        return -EIO;
    }

    if (pci_request_regions(dev, DRIVER_NAME)) {
        pci_disable_device(dev);
        return -EBUSY;
    }

    void __iomem *bar0 = pci_iomap(dev, 0, 0);  /* BAR0 映射 */
    if (!bar0) {
        pci_release_regions(dev);
        return -ENOMEM;
    }

    return 0;
}

/* remove - 驱动卸载或设备拔除时调用 */
static void remove(struct pci_dev *dev)
{
    printk(KERN_INFO "%s: remove() called\n", DRIVER_NAME);
    pci_release_regions(dev);
    pci_disable_device(dev);
}

/* pci_driver 描述符（include/linux/pci.h:1675）*/
static struct pci_driver my_pci_driver = {
    .name     = DRIVER_NAME,
    .id_table = ids,
    .probe    = probe,
    .remove   = remove,
    /* 还有：.err_handlers, .drvops, .shutdown */
};

/* 模块入口（insmod 时调用）*/
static int __init my_init(void)
{
    return pci_register_driver(&my_pci_driver);
}

/* 模块出口（rmmod 时调用）*/
static void __exit my_exit(void)
{
    pci_unregister_driver(&my_pci_driver);
}

module_init(my_init);
module_exit(my_exit);
MODULE_LICENSE("GPL");
```

## B. 为什么这样设计

### 为什么驱动和设备要分开注册？

PCIe 设备是硬件，驱动是软件，内核不知道系统里有哪些设备（取决于主板和插了什么卡）。所以：

1. **PCI 子系统先扫描设备**（`pci_scan_bus`），把所有 `pci_dev` 加入设备链表
2. **驱动注册自己**（`pci_register_driver`），带上 `id_table`
3. **匹配成功**则调用 `probe`

这种"设备先行、驱动后连"的机制允许不重启系统就动态加载驱动（`modprobe`）。

### 为什么 BAR 分配在内核 probe 阶段？

BIOS 阶段可能没有给所有 BAR 分配地址（特别是 Linux 自己接管 PCI 总线时）。Linux 重新扫描时需要重新分配，这是 PCI 子系统重新初始化的核心。

### `pci_enable_device` 做了什么？（`drivers/pci/pci.c:1981`）

```c
do_pci_enable_device(dev):
  1. pci_set_power_state(dev, PCI_D0)    ← 把设备置入 D0 电源状态
  2. pcie_aspm_powersave_config_link()   ← 配置上游 Bridge 的 ASPM
  3. pcibios_enable_device()             ← 平台特定使能（x86 BIOS callback）
  4. pci_fixup_device()                   ← 调用固件修复钩子
  5. 如果不用 MSI/MSI-X → 读 INTx 引脚
```

## C. 常见错误案例

### 错误 1：probe 里没调用 pci_enable_device，BAR 访问返回全 F

**现象**：读取 BAR 映射地址，总是返回 0xFFFFFFFF。

**排查**：确认 probe 入口调用了 `pci_enable_device(dev)`。

### 错误 2：pci_iounmap 用了错误的 bar 地址

**现象**：remove 时 kernel panic。

**排查**：pci_iounmap 的地址必须和 pci_iomap 返回值完全一致。

### 错误 3：id_table 没有加 MODULE_DEVICE_TABLE，modprobe 找不到驱动

**现象**：驱动加载了，但设备不自动匹配 probe。

**排查**：确认有 `MODULE_DEVICE_TABLE(pci, ids)` 宏。

## D. 决策树

```
驱动加载 → pci_register_driver()
  → id_table 和 pci_dev 匹配
      → probe() 被调用
          → pci_enable_device(dev) ✓
          → pci_request_regions() ✓
          → pci_iomap(dev, bar) ✓
          → 正常访问设备

      → 不匹配 → 设备留在"无驱动"状态

设备移除 → pci_remove_bus_device()
  → 驱动 remove() 被调用
      → pci_iounmap()
      → pci_release_regions()
      → 资源释放
```
