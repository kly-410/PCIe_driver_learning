# S2_01_实验 — PCI 子系统初始化 + 内核编程入门

---

## 实验 1：PCI 设备树读取

**目标**：通过 /sys/bus/pci/devices/ 理解 PCI 子系统探测到的设备树。

**步骤**：
```bash
### 查看所有 PCI 设备（用 sysfs）
ls /sys/bus/pci/devices/

### 查看某个设备的详细信息（如 RC 00:00.0）
cat /sys/bus/pci/devices/0000:00:00.0/vendor
cat /sys/bus/pci/devices/0000:00:00.0/device
cat /sys/bus/pci/devices/0000:00:00.0/subsystem_vendor
cat /sys/bus/pci/devices/0000:00:00.0/subsystem_device

### 查看 01:00.0（第一个 EP）
ls /sys/bus/pci/devices/0000:01:00.0/
cat /sys/bus/pci/devices/0000:01:00.0/vendor
```

**验收标准**：
- [ ] 能读懂 sysfs 下的设备属性文件
- [ ] RC 的 Vendor ID = 0x0000（RC 通常没有 Vendor ID）

## 实验 2：BAR 地址分配读取

**目标**：读取每个设备的 BAR 地址分配，理解内核分配结果。

**步骤**：
```bash
### 查看 01:00.0 的所有 BAR（resource0 - resource5）
cat /sys/bus/pci/devices/0000:01:00.0/resource

### 格式：start end flags（都是十六进制）
### 示例：0xf7c00000 0xf7c03fff 0x12000004

### 对比 lspci 输出
lspci -nn -s 01:00.0 | grep "Memory"

### 如果 BAR 未分配，resource 文件对应行全是 0
```

**验收标准**：
- [ ] 能解读 resource 文件的格式（start-end-flags）
- [ ] RC 的 BAR 和 EP 的 BAR 完全不同（RC 通常没有可预取 BAR）

## 实验 3：最小内核模块编译

**目标**：编译并加载一个最小 PCI 驱动模块，理解 init/probe/remove 的调用时机。

**前置**：安装内核模块编译环境：
```bash
sudo apt install linux-headers-$(uname -r) -y
```

**步骤**：
```bash
cd ~/work/code/PCIe_driver_learning/06_S2_PCI子系统初始化/

### 查看 Makefile
cat Makefile

### 编译
make

### 查看生成的 ko 文件
ls *.ko

#### 加载驱动（需要 root）
sudo insmod S2_01_pci_skeleton.ko

### 查看内核日志
dmesg | tail -20

### 卸载驱动
sudo rmmod S2_01_pci_skeleton

### 再看内核日志
dmesg | tail -10
```

**预期结果**：insmod 后 dmesg 出现 "pci: probe called"，rmmod 后出现 "pci: remove called"。

**验收标准**：
- [ ] 模块编译成功（无 warning）
- [ ] insmod 后 dmesg 有 probe 输出
- [ ] rmmod 后 dmesg 有 remove 输出

## 附：sysfs PCI 路径说明

| 路径 | 内容 |
|------|------|
| `/sys/bus/pci/devices/` | 所有 PCI 设备（符号链接到 ../devices/）|
| `/sys/bus/pci/drivers/` | 已注册的内核驱动 |
| `/sys/devices/` | 设备模型树（按拓扑结构）|
| `/proc/bus/pci/` | 传统 PCI config space 访问（不推荐）|

### 改错题（Challenge）

代码中故意埋了一个 bug，运行或审查后找出问题。

**实验代码**（`S2_01_pci_skeleton.c`，在 `06_S2_01_PCI子系统初始化/` 目录下）：

```c
/* S2_01_pci_skeleton.c —— 故意埋了 module_init/probe bug */
#include <linux/pci.h>
#include <linux/module.h>

#define VENDOR_ID 0x1d0f  /* 模拟 ID */
#define DEVICE_ID 0xdead

static int my_probe(struct pci_dev *dev, const struct pci_device_id *id)
{
    /* 缺少返回值！probe 函数没有显式 return 0 */
    printk(KERN_INFO "pci: probe called for %s\n", pci_name(dev));
    /* 如果没有 return 语句，部分编译器会返回未定义的值 */
}

static void my_remove(struct pci_dev *dev)
{
    printk(KERN_INFO "pci: remove called\n");
}

/* 错误拼写：module_init 写成 moduel_init */
module_init(my_probe);   /* ← 拼写错误：应该是 module_init */

static struct pci_driver my_pci_driver = {
    .name = "my_pci_skeleton",
    .id_table = NULL,  /* 匹配所有设备 */
    .probe = my_probe,
    .remove = my_remove,
};

/* 注册驱动 */
static int __init skeleton_init(void)
{
    return pci_register_driver(&my_pci_driver);
}
module_init(skeleton_init);   /* ← 这个拼写是对的，但上面那个是错的 */

/* 卸载驱动 */
static void __exit skeleton_exit(void)
{
    pci_unregister_driver(&my_pci_driver);
}
module_exit(skeleton_exit);

MODULE_LICENSE("GPL");
```

**问题**：

1. `module_init(my_probe)` 的拼写和用途都不对——`module_init` 应该是 `module_init(函数名)` 来注册模块加载函数，而不是把 `my_probe`（PCI probe 回调）注册为模块 init 函数
2. `my_probe` 函数缺少显式 `return 0`，导致返回不确定值

1. 找出错误的两处
2. 说明为什么 `module_init(my_probe)` 是错误的

> 提示：`module_init()` 是 Linux 模块入口点，参数是模块加载时要执行的函数。PCI 驱动的 probe 回调是 `pci_driver.probe`，不是模块 init 函数。

预期答案：有两处错误：
1. `module_init(my_probe)` 应删除（这是一行废弃的错误代码）；真正的模块 init 函数是 `skeleton_init`，已经正确地用 `module_init(skeleton_init)` 注册了
2. `my_probe` 函数末尾缺少 `return 0;`，导致内核收到不确定的返回值
