## A. 原理（是什么）

### 配置空间是什么？

PCIe 配置空间是每个设备的一组标准化寄存器，用于：
- 识别设备（Vendor ID、Device ID）
- 汇报能力（Capability）
- 配置地址空间（BAR）
- 管理中断（MSI Cap）
- 报告错误（AER）

**PCIe 配置空间大小**：
- 标准配置空间：前 256 字节（PCI 遗留，所有设备都有）
- PCIe 扩展配置空间：0x100-0xFFF（256B-4KB，仅 PCIe 设备有）

### ECAM 机制

早期 PCI 用 IO 端口访问配置空间（`in/out` 指令），慢且受限。PCIe 用 ECAM：将配置空间映射到内存地址，CPU 直接读写内存即可访问设备配置。

**内核实现**：`drivers/pci/ecam.c` + `include/linux/pci-ecam.h`

源码（`drivers/pci/ecam.c:27`）：
```c
struct pci_config_window *pci_ecam_create(struct device *dev,
        struct resource *cfgres,   // ECAM 基址和大小（从 MCFG 表获取）
        struct resource *busr,     // 总线范围
        const struct pci_ecam_ops *ops)
{
    unsigned int bus_shift = ops->bus_shift;
    struct pci_config_window *cfg;

    cfg = kzalloc_obj(*cfg);   // 分配窗口结构
    if (!cfg)
        return ERR_PTR(-ENOMEM);

    if (!bus_shift)
        bus_shift = PCIE_ECAM_BUS_SHIFT;  // 默认 20（见下）

    cfg->ops = ops;
    cfg->busr.start = busr->start;
    // ioremap ECAM 物理地址到虚拟地址
    cfg->win = ioremap(cfg->res.start, resource_size(&cfg->res));
    // ...
}
```

源码（`include/linux/pci-ecam.h:59`）：
```c
struct pci_config_window {
    struct resource     res;           // ECAM 地址范围（物理地址）
    struct resource     busr;          // 总线号范围
    unsigned int        bus_shift;    // 总线偏移量（通常=20）
    void __iomem       *win;          // 映射后的虚拟地址
    struct device      *parent;        // 所属 PCI host bridge
    const struct pci_ecam_ops *ops;   // 平台特定操作函数
};

// ECAM 地址计算公式：
// 虚拟地址 = cfg->win + (bus << bus_shift) + (dev << 15) + (func << 12) + offset
// 其中 bus_shift = 20（4KB 每总线）
```

**ECAM 映射公式**：
```
配置空间地址 = ECAM_Base + (Bus << 20) + (Dev << 15) + (Func << 12) + RegisterOffset
```

| 参数 | 含义 |
|------|------|
| ECAM_Base | RC 分配的基址（通常是 0xef800000） |
| Bus | 总线号（8位，0-255）|
| Dev | 设备号（5位，0-31）|
| Func | 功能号（3位，0-7）|
| RegisterOffset | 配置寄存器偏移（字节，0-255）|

**计算示例**：访问 Bus 1, Dev 3, Func 2 的 Vendor ID（偏移 0x00）：
```
地址 = 0xef800000 + (1 << 20) + (3 << 15) + (2 << 12) + 0x00
    = 0xef800000 + 0x00100000 + 0x00018000 + 0x00002000 + 0x00
    = 0xefa98000
```

### Type 0 和 Type 1 配置头部

PCIe 设备有两种配置空间头部：

**Type 0（Endpoint）**：
```
Offset 0x00: Vendor ID
Offset 0x02: Device ID
Offset 0x04: Command
Offset 0x06: Status
Offset 0x08: Revision ID
Offset 0x09: Class Code
Offset 0x0A: Cache Line Size
Offset 0x0B: Latency Timer
Offset 0x0C: Header Type（= 0x00）
Offset 0x0D: BIST
Offset 0x10-0x27: BAR[0-5]（6个 BAR）
Offset 0x28: Cardbus CIS Pointer
Offset 0x2C: Subsystem Vendor ID
Offset 0x2E: Subsystem ID
Offset 0x30: Expansion ROM Base Address
Offset 0x34: Capability Pointer（第一个 Capability 偏移）
Offset 0x3C: Interrupt Pin
Offset 0x3D: Interrupt Line
```

**Type 1（RC/Bridge/Switch Port）**：
- Offset 0x00-0x0F 与 Type 0 相同
- Offset 0x18: Primary Bus Number（上游总线号）
- Offset 0x19: Secondary Bus Number（下游总线号）
- Offset 0x1A: Subordinate Bus Number（最远下游总线号）
- Offset 0x1C-0x1D: I/O Base/Limit（IO 空间范围）
- Offset 0x1E-0x1F: Memory Base/Limit（Memory 空间范围）
- Offset 0x20-0x27: Prefetchable Memory Base/Limit64

### Capability 链表遍历

配置空间中的 Capability 分两类：

**标准 Capability（0x00-0xFF）**：
- 从 `PCI_CAPABILITY_LIST`（= 0x34）开始读
- 每个 Capability 是 16-bit：`低8位 = ID`，`高8位 = Next Pointer`
- 结束条件：`Cap ID == 0xFF` **或** `Next Pointer 指向的位置 < 64`（pos < 64 时提前退出，等价于 Next Pointer == 0x00；源码：`pci.h:155`）
- 典型 Cap：MSI（0x05）、Power Management（0x01）、PCI Express（0x10）

**扩展 Capability（Extended Capability，0x100-0xFFF）**：
- 从 `PCI_CFG_SPACE_SIZE`（= 0x100）开始读
- 每个 Ext Cap 是 32-bit：低16位 = Cap ID，高16位 = Next Pointer
- 结束条件：`Header == 0x00000000`（即 Next Pointer = 0）
- 典型 Ext Cap：VC（0x0002）、ARI（0x000E）、LTR（0x0018）、**SECPCI（0x0019）**、AER（0x0001）、DPC（0x001D）、SR-IOV（0x0010）

**源码依据**（`drivers/pci/pci.h:173`）：
```c
`#define PCI_FIND_NEXT_EXT_CAP(read_cfg, start, cap, prev_ptr, args...)`
us16 __pos = (start) ?: PCI_CFG_SPACE_SIZE;  // 从 0x100 开始
// ...
while (__ttl-- > 0 && __pos >= PCI_CFG_SPACE_SIZE) {
    read_cfg##_dword(args, __pos, &__header);
    if (__header == 0)   // ← Header=0 表示链表结束
        break;
    if (PCI_EXT_CAP_ID(__header) == (cap)) { /* 找到 */ }
    __pos = PCI_EXT_CAP_NEXT(__header);  // 跳到下一个 Ext Cap
}
```

**用户实际案例（GTX 1060 Mobile）**：
```
### 标准 Capability 链表
0x60: ID=0x01 (PM),       Next=0x68
0x68: ID=0x05 (MSI),      Next=0x78
0x78: ID=0x10 (PCIe),   Next=0x00 → 链表结束（Next=0x00 < 64，退出）

### Extended Capability 链表
0x100: CapID=0x0002 (VC, Virtual Channel),    Next=0x250
0x150: CapID=0x0e01 (ARI, v1),                Next=0x150
0x1a0: Header=0x00000000 → 链表结束

### Secondary PCIe Ext Cap（0x900）
setpci -s 01:00.0 900.l → Capabilities: [900 v1] Secondary PCI Express
  LnkCtl3: LnkEquIntrruptEn- PerformEqu-
  LaneErrStat: 0
→ CapID=0x0019（SECPCI，Secondary PCIe），链路均衡控制，用于链路调优和错误状态监控
```

### BAR（Base Address Register）

BAR 的作用：设备通告自己需要多少地址空间，OS 分配地址后写入 BAR，设备才知道自己的地址范围。

**BAR 位定义**：
```
Bit[0]:   0 = Memory Space, 1 = I/O Space（Memory 时为 0）
Bit[2:1]: 00 = 32-bit BAR, 10 = 64-bit BAR
Bit[3]:   Prefetchable（可预取）
Bit[31:4]: 基址（4KB 对齐起始）
```

**64-bit BAR**：BAR[n] 和 BAR[n+1] 联合表示一个 64 位地址（BAR[n+1] 必须是下一个 BAR）。

## B. 为什么这样设计

### 为什么需要 ECAM 而不是 IO 端口访问？

IO 端口访问（x86 `in/out` 指令）速度慢，且 x86 IO 空间有限（64KB）。ECAM 将配置空间映射到内存地址，CPU 用标准内存访问指令即可，效率高得多。

### 为什么 Type 0 和 Type 1 分开？

Type 0 EP 只需要配置自己的 BAR 和能力，不需要路由信息。Type 1 Bridge/Switch Port 需要记录下游总线号范围、IO/Memory 范围，用于路由决策。两个类型分开，头部更简洁。

### BAR 的 Memory Space vs I/O Space 区别？

- **Memory Space**：内存地址空间，支持预取，可被 CPU Cache 缓存，64-bit 寻址
- **I/O Space**：x86 专用（传统的 IO 端口地址空间），不被 CPU Cache，32-bit 寻址（PCIe 已基本废弃 I/O Space）

现代 PCIe 设备只用 Memory Space BAR。

## C. 常见错误案例

### 错误 1：BAR 地址未分配，设备访问返回全 F

**现象**：读写 BAR 映射的地址，总线返回全 F（0xFFFFFFFF）。

**排查**：在驱动 probe 里检查 BAR 是否已分配（`pci_resource_start(dev, bar)` 是否为 0）。

### 错误 2：64-bit BAR 用了不对齐的地址

**现象**：驱动分配了 4GB 内存，但 BAR 只报告需要 2GB。

**排查**：64-bit BAR 需要 8-byte 对齐（实际上 4KB 对齐）；检查 BAR 解析代码是否正确处理 64-bit BAR 的高位部分。

### 错误 3：Capability 链表遍历——混淆了标准 Cap 和 Ext Cap 的结束条件

**现象**：驱动加载时找不到 Ext Cap，或者把 Ext Cap 的 Header=0 当成错误。

**根因**：标准 Capability 和 Extended Capability 是两套独立的链表，结束条件不同：

| | 标准 Capability（0x00-0xFF）| 扩展 Capability（0x100-0xFFF）|
|:---|:---|:---|
| **结束条件** | `Cap ID == 0xFF` **或** `Next Pointer < 64`（pos < 64，提前退出） | `Header == 0x00000000`（Next Offset = 0）|
| **遍历起点** | 0x34 | 0x100 |
| **每次读** | 16-bit | 32-bit |
| **源码依据** | `PCI_FIND_NEXT_CAP`（pci.h:137）| `PCI_FIND_NEXT_EXT_CAP`（pci.h:173）|

**排查**：
- 标准 Cap 遍历失败 → 检查 `PCI_STATUS` 的 `CAP_LIST` 位是否置位
- Ext Cap 遍历失败 → 检查 `dmesg` 有无 `cannot find ext cap"` 或读到的 Header 是否为 0

## D. 决策树

```
访问 PCIe 配置空间
├── 知道 Bus/Dev/Func
│   └── ECAM 公式计算地址 = ECAM_Base + (Bus<<20) + (Dev<<15) + (Func<<12) + offset
│
├── 判断设备类型
│   ├── Header Type = 0x00 → Type 0（EP）
│   └── Header Type = 0x01 → Type 1（Bridge/Switch Port）
│
├── 找 Capability
│   ├── 目标在 0x00-0xFF（标准 Capability）
│   │   └── 从 0x34 开始，满足任一则结束：
│   │       • Capability ID == 0xFF
│   │       • Next Pointer < 64（即 pos < PCI_STD_HEADER_SIZEOF）
│   └── 目标在 0x100-0xFFF（扩展 Capability）
│       └── 从 0x100 开始，`Header == 0x00000000` → 链表结束
│
│
└── 解析 BAR
    ├── Bit[0] = 0 → Memory Space
    │   ├── Bit[2:1] = 00 → 32-bit BAR
    │   └── Bit[2:1] = 10 → 64-bit BAR（BAR[n] + BAR[n+1]）
    └── Bit[0] = 1 → I/O Space（基本废弃）
```
